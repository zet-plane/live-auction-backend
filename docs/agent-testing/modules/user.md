# 用户模块测试说明

## 1. 模块目标

用户模块负责账号密码注册、登录、JWT 鉴权、当前用户资料查询、资料修改和当前账号注销。

根据当前代码，用户模块同时为其他需要登录的模块提供 `AuthenticateToken` 鉴权能力，并用 `identity` 区分普通用户和商家。

## 2. 测试边界

Agent 可以测试：

- 注册接口 `POST /api/v1/auth/register`。
- 登录接口 `POST /api/v1/auth/login`。
- 获取当前用户接口 `GET /api/v1/users/me`。
- 修改当前用户接口 `PUT /api/v1/users/me`。
- 注销当前用户接口 `DELETE /api/v1/users/me`。
- `Authorization: Bearer <token>` 鉴权中间件行为。
- `Register`、`Login`、`Authenticate`、`UpdateProfile`、`DeleteMe` Service 逻辑。
- `User` 模型字段、账号唯一性、密码存储、软删除行为。
- JWT 签名、过期、篡改和用户不存在时的认证结果。
- HTTP 统一响应结构和错误码。

当前代码未看到用户模块使用 Redis、WebSocket 或外部第三方服务；这些依赖不属于用户模块测试边界。

## 3. 禁止事项

- 不测试支付、订单、物流、竞拍出价或竞拍场次状态流转。
- 不调用真实短信、邮件、支付或其他第三方服务。
- 不直接清空数据库。
- 不修改生产配置或复用线上真实用户账号。
- 不把测试账号用于本次测试以外的业务数据。
- 不在测试报告中写入真实 token、密码、数据库地址或可复用凭据。
- 不绕过业务接口直接修改用户状态，除非文档明确要求用于故障注入。
- 本地单元测试不允许直接连接数据库，必须使用 mock/fake store。
- Agent 连接线上或线上等价数据库时，只能操作本次测试创建的数据或带测试批次 ID 的数据。

## 4. 业务规则

- 注册请求包含 `account` 和 `password`。
- 注册成功后创建新用户，默认 `identity` 为 `user`。
- 注册成功后返回 token 和用户信息。
- `account` 在数据库中要求全局唯一。
- `password` 不允许出现在用户响应 DTO 中。
- 密码存储时必须是哈希结果，不能明文落库。
- 登录成功后返回 token 和用户信息。
- 登录账号不存在或密码错误时应返回未授权错误。
- 需要登录的接口必须携带 `Authorization: Bearer <token>`。
- token 签名错误、格式错误、过期或对应用户不存在时应认证失败。
- 当前用户资料允许修改 `name`、`avatar_url`、`motto`、`identity`。
- `identity` 只允许 `user` 或 `merchant`。
- 注销当前用户后，该用户不能继续通过旧 token 访问登录态接口。

根据当前代码结构推断：

- 注销通过 GORM `DeletedAt` 执行软删除。
- 用户可以通过修改资料接口把 `identity` 从 `user` 改为 `merchant`。

## 5. 业务不变量

- 同一个 `account` 不能存在两个有效用户。
- 响应中的用户信息不能包含 `password`。
- 数据库中保存的密码不能等于注册或登录时提交的明文密码。
- 未认证请求不能访问 `/api/v1/users/me` 下的接口。
- 无效 token 不能通过认证。
- 已注销用户不能继续被认证为当前用户。
- `identity` 不能被更新为 `user` 和 `merchant` 以外的值。

不变量失败时，agent 除常规失败报告外，必须额外输出：

```text
违反的不变量：<不变量名称>
违反位置：<模块/接口/步骤编号>
期望状态：
实际状态：
```

## 6. 测试数据准备

需要准备：

- 至少 1 个普通用户注册请求。
- 至少 1 个用于重复注册的相同账号。
- 至少 1 个错误密码登录请求。
- 至少 1 个有效 token。
- 至少 1 个格式错误 token。
- 至少 1 个签名被篡改 token。
- 至少 1 个过期 token。
- 至少 1 个带测试批次 ID 的账号前缀，例如 `agent_user_<batch>_alice`。

如果执行接口契约、模块集成或状态一致性测试，测试账号必须可识别并在测试结束后验证清理或软删除结果。

## 7. 依赖策略建议

| 测试类型 | 依赖策略 | 原因 |
| --- | --- | --- |
| 本地单元测试 | 使用 fake store、固定 token secret、固定时间；禁止直连数据库 | 稳定验证注册、登录、鉴权、资料更新和注销逻辑 |
| Agent 接口契约测试 | 使用真实 handler 或本地服务；允许连接测试数据库；不依赖 Redis 或 WebSocket | 验证真实请求绑定、响应结构、错误码和鉴权中间件 |
| Agent 模块集成测试 | 使用真实 GORM store 和测试数据库 | 验证账号唯一索引、软删除、字段持久化和自动迁移 |
| Agent 并发测试 | 使用真实数据库唯一约束和真实 HTTP 并发请求 | 验证重复注册、并发更新和注销/查询竞态 |
| 状态一致性测试 | 对比 HTTP 响应、查询接口和数据库记录 | 验证接口返回与持久化状态一致 |

## 8. 单元测试

需要覆盖：

- 注册时账号去除首尾空格。
- 注册成功生成默认昵称、默认 `identity=user` 和非空 token。
- 注册时密码被哈希保存。
- 重复账号注册失败。
- 空账号、超长账号、空密码注册失败。
- 登录成功返回 token 和用户信息。
- 登录错误密码返回未授权错误。
- 登录不存在账号返回未授权错误。
- token 签名和验证。
- token 过期后认证失败。
- token 被篡改后认证失败。
- token 对应用户不存在时认证失败。
- 修改资料时字段被去除首尾空格。
- 修改 `identity` 为 `user` 或 `merchant` 成功。
- 修改非法 `identity` 失败。
- 注销不存在用户返回 not found。
- 注销成功后 store 不再能查询到该用户。

## 9. 接口契约测试

需要覆盖：

- `POST /api/v1/auth/register` 成功响应结构：`code`、`message`、`data.token`、`data.user`。
- 注册响应中的 `data.user` 不包含 `password`。
- 注册请求缺少 `account` 或 `password` 时返回参数错误。
- 注册请求 `account` 长度小于 3 或大于 64 时返回参数错误。
- 注册请求 `password` 长度小于 6 或大于 72 时返回参数错误。
- 重复账号注册失败响应。
- `POST /api/v1/auth/login` 成功响应结构。
- 登录错误密码和不存在账号的失败响应。
- `GET /api/v1/users/me` 成功返回当前用户资料。
- `GET /api/v1/users/me` 未携带 token、错误 Bearer 格式、无效 token 时返回未授权。
- `PUT /api/v1/users/me` 成功响应 `data=null`。
- `PUT /api/v1/users/me` 非法字段长度和非法 `identity` 的失败响应。
- `DELETE /api/v1/users/me` 成功响应 `data=null`。
- 注销后旧 token 再访问 `/api/v1/users/me` 返回未授权。

## 10. 业务场景测试

### 注册后登录并查询当前用户

Given:

- 测试批次账号不存在。
- 注册请求包含合法账号和密码。

When:

- 调用注册接口。
- 使用同一账号密码调用登录接口。
- 使用登录返回的 token 调用当前用户查询接口。

Then:

- 注册和登录都返回 token。
- 查询接口返回的 `id`、`account`、`identity` 与登录响应一致。
- 响应中不包含 `password`。
- 数据库中密码不是明文。

### 修改资料并升级为商家身份

Given:

- 已注册并登录的普通用户。

When:

- 调用 `PUT /api/v1/users/me` 修改 `name`、`avatar_url`、`motto`、`identity=merchant`。
- 再次调用 `GET /api/v1/users/me`。

Then:

- 修改接口成功。
- 查询接口返回最新资料。
- `identity` 为 `merchant`。
- 数据库记录与查询接口一致。

### 注销后旧 token 失效

Given:

- 已注册并登录的用户。
- 已获得有效 token。

When:

- 调用 `DELETE /api/v1/users/me`。
- 使用同一个 token 再次调用 `GET /api/v1/users/me`。

Then:

- 注销接口成功。
- 后续查询返回未授权。
- 数据库中该用户为已删除状态或无法通过普通查询查到。

## 11. 异常测试

需要覆盖：

- 注册账号为空。
- 注册账号只有空格。
- 注册账号长度超过 64。
- 注册账号长度小于接口绑定要求。
- 注册密码为空。
- 注册密码只有空格。
- 注册密码长度小于接口绑定要求。
- 注册密码长度超过接口绑定要求。
- 重复注册同一账号。
- 登录账号不存在。
- 登录密码错误。
- 登录账号或密码为空。
- 无 `Authorization` header。
- `Authorization` 不是 `Bearer <token>` 格式。
- token 分段数量错误。
- token 签名错误。

## 12. 边界测试

需要覆盖：

- `account` 长度刚好 3。
- `account` 长度刚好 64。
- `account` 长度为 65。
- `password` 长度刚好 6。
- `password` 长度刚好 72。
- `password` 长度为 73。
- `name` 长度刚好 1。
- `name` 长度刚好 64。
- `name` 长度为 65。
- `avatar_url` 长度刚好 512。
- `avatar_url` 长度为 513。
- `motto` 长度刚好 255。
- `motto` 长度为 256。
- token 当前时间刚好早于过期时间。
- token 当前时间刚好等于过期时间。
- token 当前时间晚于过期时间。

## 13. 并发测试

需要覆盖：

- 多个请求同时注册同一个账号。
- 同一用户同时多次登录。
- 同一用户同时修改不同资料字段。
- 注销和查询当前用户同时发生。
- 注销和修改资料同时发生。

验证要求：

- 同一账号最终只能有一个有效用户。
- 并发重复注册最多只能有一个成功响应。
- 并发登录不能改变用户资料或密码。
- 注销完成后不能继续认证成功。
- 并发修改产生的最终资料必须与最后一次成功写入一致，不能出现部分字段损坏。

如果业务需要定义“并发修改最后写入者”的精确判定方式，agent 必须先询问用户。

## 14. 状态一致性测试

需要验证：

- 注册响应中的用户信息与数据库用户记录一致。
- 登录响应中的用户信息与数据库用户记录一致。
- 查询当前用户响应与数据库用户记录一致。
- 修改资料后查询接口与数据库记录一致。
- 注销接口成功后，普通查询无法继续查到该用户。
- HTTP 错误响应的 `code`、`message` 与错误类型一致。

状态不一致时，agent 必须记录：

- 哪两个数据源不一致。
- 哪个接口或步骤触发不一致。
- 不一致是否会影响认证、商家权限或后续竞拍流程。

## 15. WebSocket 测试

不适用。

当前用户模块未定义 WebSocket 连接、事件或推送消息。用户身份对 WebSocket 鉴权的影响如需测试，应在具体 WebSocket 模块或跨模块流程文档中定义。

## 16. 回归测试

以下问题一旦出现，必须沉淀为回归测试：

- 注册或登录响应泄露 `password`。
- 密码明文落库。
- 重复账号并发注册产生多个有效用户。
- 错误密码可以登录。
- 过期 token 仍可认证成功。
- 被篡改 token 仍可认证成功。
- 已注销用户旧 token 仍可访问登录态接口。
- 非法 `identity` 被保存。
- 接口绑定规则与 Service 规则不一致导致不同入口行为不一致。

## 17. 通过标准

**核心验证点（全部通过才算过）：**

- 注册、登录、查询、修改、注销接口响应结构符合统一响应格式，并有 HTTP 响应作为证据。
- 注册成功后数据库存在一个有效用户，并有数据库查询结果作为证据。
- 密码不明文落库，并有数据库字段或单元测试断言作为证据。
- 响应 DTO 不包含 `password`，并有 HTTP 响应或 DTO 单元测试作为证据。
- 重复账号注册被拒绝，并有接口响应或 Service 单元测试作为证据。
- 无效登录和无效 token 被拒绝，并有错误响应或单元测试作为证据。
- 资料修改后 HTTP 查询结果与数据库记录一致，并有两边状态作为证据。
- 注销后旧 token 不能继续访问登录态接口，并有后续请求响应作为证据。

**辅助验证点（建议验证，可附说明跳过）：**

- token 过期时间符合配置 `auth.token_ttl` 或默认 24 小时。
- 用户模块自动迁移成功创建或更新用户表结构。
- 错误响应中的业务错误码与 `pkg/errorx` 定义一致。

## 18. 需用户确认的问题

- 注册和登录的 Service 层是否也必须严格执行接口绑定中的长度规则：`account` 最小 3、`password` 最小 6 最大 72。
- 注销是否必须保持 GORM 软删除语义，还是未来允许物理删除。
- 普通用户是否允许自行把 `identity` 修改为 `merchant`，是否需要额外审核或后台权限。
- 并发资料修改时是否采用最后写入者覆盖，还是需要版本号或冲突检测。
- 账号大小写是否敏感，例如 `Alice` 和 `alice` 是否允许同时存在。

```text

复现步骤：

实际结果：
相关证据：

已知缺口：（本次测试因文档或实现原因未覆盖的风险，以及建议如何补充）
```

如果是不变量违反，额外输出：

```text

违反位置：
期望状态：
实际状态：
```
