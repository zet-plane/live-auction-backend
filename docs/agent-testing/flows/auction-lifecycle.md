# 竞拍全生命周期测试说明

## 1. 测试目标

验证直播竞拍从准备到成交的完整闭环，确保跨模块协作后的最终业务结果正确。

## 2. 覆盖模块

本流程覆盖：

- 用户模块。
- 商家模块。
- 拍品模块。
- 竞拍规则模块。
- 竞拍场次模块。
- 出价模块。
- 排名模块。
- WebSocket 推送，如果当前实现已支持。
- 成交结果。

## 3. 不覆盖范围

- 支付。
- 订单履约。
- 物流。
- 真实短信。
- 真实第三方服务。
- 大规模压测。
- AI 氛围生成和动态定价策略。

## 4. 前置条件

需要具备：

- 可启动的后端服务。
- 可用的测试数据库。
- 可用的 Redis，如果当前实现依赖 Redis。
- 可创建测试用户和商家的接口或测试夹具。
- 可创建拍品、规则和场次的接口或测试夹具。

## 5. 测试数据

建议准备：

- 商家 S。
- 用户 A。
- 用户 B。
- 拍品 P。
- 竞拍规则 R：
  - 起拍价 1000。
  - 最小加价 100。
  - 场次开始时间为当前可测试时间。
  - 场次结束时间晚于开始时间。
- 竞拍场次 Session。

## 6. 主流程

### 完整成交链路 <!-- id: auction-complete-settlement -->

Given:

- 商家 S 已存在。
- 用户 A 和用户 B 已存在。
- 拍品 P 已创建。
- 竞拍规则 R 已配置。
- 竞拍场次 Session 已创建。

When:

1. 开始竞拍场次。
2. 用户 A 出价 1100。
3. 用户 B 出价 1200。
4. 查询当前排名。
5. 触发落锤。
6. 查询成交结果。
7. 用户 A 尝试再次出价 1300。

Then:

- 用户 A 第一次出价成功。
- 用户 B 出价成功。
- 当前价为 1200。
- 排名第一为用户 B。
- 成交用户为用户 B。
- 成交价为 1200。
- 落锤后用户 A 再次出价失败。
- 成交结果未被后续失败出价改变。

## 7. 异常流程 <!-- id: auction-error-cases -->

需要覆盖：

- 拍品不存在时创建场次失败。
- 未配置规则时创建或开始场次失败。
- 未开始场次出价失败。
- 已结束场次出价失败。
- 无人出价时落锤不生成虚假成交。
- 重复落锤不产生多个成交结果。

## 8. 并发流程 <!-- id: auction-concurrent-bids -->

需要覆盖：

- 用户 A 和用户 B 同时出价。
- 多个用户以不同价格同时出价。
- 多个用户以相同价格同时出价。
- 落锤和出价同时发生。

验证要求：

- 最终成交结果唯一。
- 当前价和成交价一致。
- 最高出价人和成交用户一致。
- 重复请求不产生重复成交。

如果落锤和出价同时发生的优先级规则未定义，agent 必须先询问用户。

## 9. 状态一致性验证

需要验证：

- HTTP 响应中的当前价、场次状态和成交结果。
- 查询接口返回的当前价、排名和成交结果。
- MySQL 中场次、出价记录和成交结果。
- Redis 中实时状态，如果当前实现依赖 Redis。
- WebSocket 推送消息，如果当前实现已支持。
- 日志中的关键状态流转。

## 10. WebSocket 验证

如果当前实现支持 WebSocket，需要验证：

- 场次开始消息。
- 出价成功消息。
- 排名变化消息。
- 落锤成交消息。
- 推送结果与最终查询结果一致。

如果当前实现未支持 WebSocket，本节标记为不适用。

## 11. 通过标准

全生命周期测试通过必须满足：

- 主流程完整执行成功。
- 出价、排名、落锤和成交结果一致。
- 落锤后不能继续有效出价。
- 重复落锤不能产生多个成交结果。
- HTTP、MySQL、Redis 和 WebSocket 中的关键状态一致。
- 所有失败场景都有明确错误响应。

## 12. 失败报告格式

测试失败时，agent 必须输出：

```text
失败场景：
复现步骤：
期望结果：
实际结果：
相关证据：
可能原因：
影响范围：
建议修复点：
建议新增的回归测试：
```
