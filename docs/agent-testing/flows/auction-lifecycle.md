# 竞拍全生命周期测试说明

## 1. 测试目标

验证直播竞拍从准备到成交的完整闭环，确保跨模块协作后的最终业务结果正确。

## 2. 覆盖模块

本流程覆盖：

- 用户模块。
- 商家模块。
- 房间模块。
- 拍品模块。
- 竞拍规则模块。
- 保证金模块。
- 出价模块。
- 排名模块。
- WebSocket 推送，如果当前实现已支持。
- 成交结果。

说明：早期设计中的 `auction-session` 已由当前设计中的 `room` 替代。本流程中的直播竞拍上下文由房间承载，拍品竞拍状态由商品模块承载。

## 3. 不覆盖范围

- 订单支付。
- 订单履约。
- 物流。
- 真实短信。
- 真实第三方服务。
- 大规模压测。
- AI 氛围生成和动态定价策略。

## 4. 前置条件

需要具备：

- 可启动的后端服务。
- 可用的测试数据库。
- 可用的 Redis，如果当前实现依赖 Redis。
- 可创建测试用户和商家的接口或测试夹具。
- 可创建直播间、拍品和规则的接口或测试夹具。
- 可通过保证金接口为本批次用户缴纳站内保证金。

## 5. 测试数据

建议准备：

- 商家 S。
- 用户 A。
- 用户 B。
- 用户 C：
  - 只连接 WebSocket 旁观。
  - 不缴纳保证金。
  - 不出价。
- 直播间 Room：
  - 初始状态为 `idle`。
  - 测试开始后切换为 `live`。
- 拍品 P1 和 P2。
- 竞拍规则 R：
  - 起拍价 1000。
  - 最小加价 100。
  - 保证金 5000。
  - 竞拍开始时间为当前可测试时间。
  - 竞拍结束时间晚于开始时间。

## 6. 主流程

### 完整成交链路 <!-- id: auction-complete-settlement -->

Given:

- 商家 S 已存在。
- 用户 A 和用户 B 已存在。
- 用户 C 已存在。
- 直播间 Room 已创建并开播。
- 拍品 P1 和 P2 已创建。
- 竞拍规则 R 已配置，且要求保证金。
- 拍品 P1 和 P2 已绑定到直播间 Room。
- 用户 A、B、C 已连接到 Room 的 WebSocket。

When:

1. 上架 P1 和 P2。
2. 验证房间待拍队列包含 P1 和 P2。
3. 开始 P1 竞拍。
4. 验证 P1 拍卖倒计时。
5. 用户 A 未缴保证金出价 1100。
6. 用户 A 缴纳保证金。
7. 用户 A 出价 1100。
8. 用户 B 未缴保证金出价 1200。
9. 用户 B 缴纳保证金。
10. 用户 B 出价 1200。
11. 用户 A 出价 1300。
12. 用户 B 出价 1400。
13. 用户 A 出价 1500。
14. 用户 B 出价 1600。
15. 查询当前排名。
16. 触发竞拍结束或过期结算。
17. 查询成交结果。
18. 用户 A 尝试再次出价 1700。

Then:

- P1 和 P2 均进入房间待拍队列，HTTP 房间详情与 Redis 队列一致。
- P1 进入 `ongoing` 后 Redis `end_time_unix` 与规则结束时间一致，HTTP `remaining_ms` 与 Redis 倒计时一致。
- 用户 A 未缴保证金出价失败，且不改变 Redis、MySQL 或 WebSocket 成功事件。
- 用户 A 缴纳保证金后出价成功。
- 用户 B 未缴保证金出价失败。
- 用户 B 缴纳保证金后出价成功。
- A、B 多轮交替出价后当前价为 1600。
- 排名第一为用户 B。
- 排名第二为用户 A。
- 成交用户为用户 B。
- 成交价为 1600。
- 用户 C 不产生保证金、出价、排行榜或成交数据。
- 用户 C 只接收房间广播，不接收只属于 A 或 B 的用户单播。
- 竞拍结束后用户 A 再次出价失败。
- 成交结果未被后续失败出价改变。

## 7. 异常流程 <!-- id: auction-error-cases -->

需要覆盖：

- 拍品不存在时创建或上架拍品失败。
- 未配置规则时创建或开始拍品竞拍失败。
- 房间未开播时的拍品流程按当前模块契约验证或记录为已知行为。
- 未开始竞拍的拍品出价失败。
- 未缴纳保证金时出价失败。
- 已结束拍品出价失败。
- 无人出价时过期结算不生成虚假成交。
- 重复结束或重复结算不产生多个成交结果。

## 8. 并发流程 <!-- id: auction-concurrent-bids -->

本次基础端到端流程测试不执行并发场景。并发出价、相同价格竞争、重复请求和竞拍结束同时出价应作为专项并发测试单独执行。

专项并发测试需要覆盖：

- 用户 A 和用户 B 同时出价。
- 多个用户以不同价格同时出价。
- 多个用户以相同价格同时出价。
- 竞拍结束和出价同时发生。

验证要求：

- 最终成交结果唯一。
- 当前价和成交价一致。
- 最高出价人和成交用户一致。
- 重复请求不产生重复成交。

如果竞拍结束和出价同时发生的优先级规则未定义，agent 必须先询问用户。

## 9. 状态一致性验证

需要验证：

- HTTP 响应中的当前价、房间状态、拍品状态和成交结果。
- 保证金接口响应和 `deposits` 表状态。
- 查询接口返回的当前价、排名和成交结果。
- MySQL 中房间、拍品、保证金、出价记录和成交结果。
- Redis 中实时状态，如果当前实现依赖 Redis。
- Redis 房间待拍队列和拍品倒计时状态。
- WebSocket 推送消息，如果当前实现已支持。
- 日志中的关键状态流转。

## 10. WebSocket 验证

如果当前实现支持 WebSocket，需要验证：

- 竞拍开始消息。
- 出价成功消息。
- 用户被超越消息，如果当前实现已支持。
- 排名变化消息。
- 竞拍结束或成交消息。
- A、B、C 均应接收房间广播。
- C 不应接收只属于 A 或 B 的用户单播。
- 推送结果与最终查询结果一致。

如果当前实现未支持 WebSocket，本节标记为不适用。

## 11. 通过标准

全生命周期测试通过必须满足：

- 主流程完整执行成功。
- 房间、拍品、保证金、出价、排名和成交结果一致。
- 多件拍品上架后的房间待拍队列一致。
- 拍卖倒计时与 Redis `end_time_unix` 一致。
- 未缴保证金不能产生有效出价。
- A、B 多轮交替出价后最高出价人和成交用户一致。
- 用户 C 只作为旁观者接收房间广播，不产生业务数据。
- 竞拍结束后不能继续有效出价。
- 重复结束或重复结算不能产生多个成交结果。
- HTTP、MySQL、Redis 和 WebSocket 中的关键状态一致。
- 所有失败场景都有明确错误响应。

## 12. 失败报告格式

测试失败时，agent 必须输出：

```text
失败场景：
复现步骤：
期望结果：
实际结果：
相关证据：
可能原因：
影响范围：
建议修复点：
建议新增的回归测试：
```
