# 模块测试文档模板

> 使用方式：复制本模板到 `docs/agent-testing/modules/<module>.md`，然后按模块实际情况填写。不适用的接口、字段、场景或测试类型写明“不适用”和原因。

通用依赖授权、证据、清理和敏感信息规则见 `docs/agent-testing/templates/protocol.md`。本模板只描述单模块测试契约结构。

本模板按“测试对象优先”组织：先说明代码定位、数据准备、接口、字段、状态和场景，再说明每个对象适用哪些测试方法。不要把文档写成一长串按“单元测试 / 接口测试 / 异常测试”分类的测试点。

## 1. 模块目标

说明这个模块负责什么、服务哪些业务流程、核心实体是什么。

## 2. 代码定位索引

列出 agent 和人工读者定位测试对象时必须看的代码。

| 对象 | 代码位置 | 说明 |
| --- | --- | --- |
| Router | `internal/app/<module>/router` | HTTP 路由注册 |
| Handler | `internal/app/<module>/handler` | 请求绑定、鉴权上下文、响应返回 |
| DTO | `internal/app/<module>/dto` | 请求字段、响应字段、绑定规则 |
| Service | `internal/app/<module>/service` | 业务规则、状态流转、依赖编排 |
| DAO | `internal/app/<module>/dao` | 持久化接口和数据库实现 |
| Model | `internal/app/<module>/model` | GORM 模型、字段、索引、软删除 |
| 单元测试建议位置 | `internal/app/<module>/service/*_test.go` | 使用 mock/fake store、cache、time、ID |
| Agent 测试契约 | `docs/agent-testing/modules/<module>.md` | 接口契约、集成、场景和一致性测试边界 |

如果模块存在 Redis、WebSocket、外部服务、后台任务或日志关键字，应增加对应行。

## 3. 测试边界

Agent 可以测试：

- HTTP 接口。
- Service 方法。
- DAO / Model。
- MySQL 表。
- Redis key。
- WebSocket 消息。
- 日志关键字。
- 与本模块直接相关的跨模块依赖。

Agent 不应测试不属于本模块的完整业务流程；需要跨模块验证时应转到 `flows/<flow>.md`。

## 4. 禁止事项

说明 agent 不允许执行的动作。

示例：

- 不允许清空数据库。
- 不允许修改生产配置。
- 不允许调用真实第三方服务。
- 不允许自行创造业务规则。
- 不允许扩大到当前模块以外的业务。
- 本地单元测试不允许直接连接数据库、Redis、HTTP 服务、WebSocket 或外部系统，必须使用 mock/fake 数据。
- Agent 连接线上或线上等价数据库、记录证据、清理数据和处理敏感信息时，遵守 `templates/protocol.md`。

## 5. 测试依赖策略

| 测试类型 | 依赖策略 | 原因 |
| --- | --- | --- |
| 本地单元测试 | 使用 mock/fake Store、Cache、外部依赖、时间和 ID；禁止直连 MySQL、Redis、HTTP 服务或 WebSocket | 稳定验证业务规则、字段归一化、错误分支和状态流转 |
| Agent 接口契约测试 | 使用真实 handler 或本地服务；使用真实测试数据库；模块依赖 Redis 时使用真实测试 Redis；第三方不可控服务 mock | 验证真实请求绑定、鉴权、响应结构、错误码和字段形状 |
| Agent 模块集成测试 | 使用真实 DAO、Service、测试数据库，必要时真实测试 Redis | 验证事务、索引、软删除、缓存协作和持久化状态 |
| 场景测试 | 使用真实接口链路和真实测试数据库；按场景要求使用真实测试 Redis 或 WebSocket | 验证用户可见业务链路和跨接口状态变化 |
| Agent 并发一致性测试 | 使用真实测试数据库、Redis 和真实 HTTP 并发请求 | mock/fake 无法证明并发控制、唯一约束和最终一致性 |
| Agent 性能压测 | 使用准生产或线上等价依赖、独立压测源和监控数据 | 验证容量、延迟、错误率、资源瓶颈和稳定性 |
| 状态一致性测试 | 对比 HTTP 响应、数据库、Redis、WebSocket 消息和日志证据 | 验证外部可见结果和内部状态一致 |

## 6. 全局测试数据准备

说明测试前需要准备的通用数据。

必须包含：

- 测试批次 ID，例如 `agent_<module>_<YYYYMMDDHHMMSS>`。
- 本模块创建数据的命名前缀。
- 所需用户身份和 token 类型。
- 所需前置业务实体。
- 合法请求体样例。
- 非法请求体集合。
- 需要验证或清理的数据库记录。
- 需要验证或清理的 Redis key。
- 清理策略或软删除验证策略。

示例：

```text
测试批次 ID：agent_<module>_<batch>
账号前缀：agent_<module>_<batch>_
数据只允许操作本批次创建的数据。
测试结束后必须记录清理结果或软删除验证结果。
```

## 7. 业务规则

用自然语言描述模块的核心业务规则。

必须区分：

- 事实：从代码、已有文档或接口定义中直接得到。
- 推断：根据命名、调用关系或已有模块模式得出，但没有明确业务文字。
- 需确认：无法从代码判断，且会影响测试边界或通过标准。

推断内容必须标记为“根据当前代码结构推断”。需确认内容必须放到“需用户确认的问题”章节。

## 8. 业务不变量

列出系统永远不能违反的规则。

示例：

- 已结束的竞拍不能继续出价。
- 成交结果一旦生成，不允许被后续出价改变。

不变量失败时，agent 除常规失败报告外，必须额外输出：

```text
违反的不变量：<不变量名称>
违反位置：<模块/接口/步骤编号>
期望状态：
实际状态：
```

## 9. 字段规则索引

字段规则索引用来避免字段测试点散落在各类测试章节中。

### <Entity 或 Request/Response 名称>

| 字段 | 来源 | 规则 | 涉及接口 / 方法 | 测试点 ID |
| --- | --- | --- | --- | --- |
| `<field>` | request/db/response | 必填、长度、枚举、归一化、默认值、脱敏等 | `<METHOD path>` | `<MODULE>.FIELD.<field>.*` |

每个重要字段至少说明：

- 来自请求、数据库、响应、Redis 还是 WebSocket。
- 是否必填。
- 长度、金额、时间、枚举、状态、默认值、归一化规则。
- 哪些接口读写该字段。
- 字段失败时应表现为参数错误、业务错误、未授权、not found 或其他错误。

## 10. 接口测试契约

每个接口使用同一张测试卡片。没有 HTTP 接口的模块可写“不适用”，并改为 Service / DAO 测试契约。

### `<METHOD> <path>` <接口名称>

#### 代码定位

| 层级 | 位置 | 说明 |
| --- | --- | --- |
| Router | `internal/app/<module>/router/...` | 路由注册 |
| Handler | `internal/app/<module>/handler/...` | 请求绑定、鉴权上下文、响应返回 |
| DTO | `internal/app/<module>/dto/...` | Request / Response 字段 |
| Service | `internal/app/<module>/service/...` | 业务规则 |
| DAO | `internal/app/<module>/dao/...` | 数据读写 |
| Model | `internal/app/<module>/model/...` | 持久化字段 |
| Cache / WebSocket / 外部依赖 | `...` | 不适用时写“不适用” |

#### 接口职责

说明这个接口负责完成什么业务动作，以及不负责什么。

#### 请求字段

| 字段 | 必填 | 规则 | 失败表现 |
| --- | --- | --- | --- |
| `<field>` | 是/否 | 长度、枚举、金额、时间、归一化、默认值 | 参数错误/业务错误/未授权/not found |

#### 响应字段

| 字段 | 规则 | 证据 |
| --- | --- | --- |
| `<field>` | 不泄露敏感字段、类型、默认值、派生值等 | HTTP 响应 |

#### 测试数据准备

- 需要的身份和 token。
- 需要的前置数据。
- 合法请求体。
- 非法请求体集合。
- 数据前缀和清理要求。

#### 成功路径

- 用可验证语言列出成功路径。
- 每条成功路径说明需要的证据。

#### 失败路径

- 未登录、身份不足、归属不匹配。
- 参数非法、字段越界、枚举非法。
- 状态不允许、资源不存在、重复提交。
- 存储或缓存失败时的行为。

#### 状态和一致性验证

- HTTP 响应与数据库状态一致。
- 数据库表之间一致。
- Redis / WebSocket / 日志与数据库一致。
- 失败时不能留下半成功状态。

#### 适用测试方法

| 测试类型 | 是否适用 | 说明 |
| --- | --- | --- |
| 本地单元测试 | 是/否 | 使用 mock/fake 验证哪些 Service 规则 |
| 接口契约测试 | 是/否 | 使用真实测试数据库验证请求和响应 |
| 模块集成测试 | 是/否 | 使用真实 DAO / Service 验证哪些状态 |
| 场景测试 | 是/否 | 是否被核心业务场景覆盖 |
| 并发一致性测试 | 是/否 | 是否需要真实并发验证 |
| 状态一致性测试 | 是/否 | 需要对比哪些证据 |

## 11. Service / DAO 测试契约

用于补充没有直接 HTTP 入口，或接口卡片无法完整表达的内部业务能力。

### `<ServiceMethod>` / `<DAOMethod>`

#### 代码定位

| 层级 | 位置 | 说明 |
| --- | --- | --- |
| Service | `internal/app/<module>/service/...` | 方法定义 |
| Store 接口 | `internal/app/<module>/dao/...` | mock/fake 契约 |
| DAO 实现 | `internal/app/<module>/dao/...` | 集成测试目标 |
| Model | `internal/app/<module>/model/...` | 持久化字段 |

#### 测试数据准备

- fake store 初始数据。
- fake cache 初始状态。
- 真实数据库集成测试需要的数据。

#### 单元测试点

- 使用 mock/fake 接口验证业务逻辑。
- 覆盖成功、失败、边界和错误传播。

#### 集成测试点

- 使用真实测试数据库验证 DAO / 事务 / 索引 / 软删除。

## 18. 通过标准

说明什么叫测试通过。

通过标准分两层：

**核心验证点（全部通过才算过）：**

- 说明哪些业务不变量必须成立。
- 说明哪些接口响应必须符合预期。
- 说明哪些存储状态必须一致。
- 说明哪些核心场景必须通过。

**辅助验证点（建议验证，可附说明跳过）：**

- 说明哪些日志格式、WebSocket 消息、缓存清理等辅助验证点。

每条通过标准必须包含可验证证据。

## 20. 失败报告格式

测试失败时，agent 必须输出：

```text
失败场景：
复现步骤：
期望结果：
实际结果：
相关证据：
可能原因：
建议修复点：
建议新增的回归测试：
已知缺口：（本次测试因文档或实现原因未覆盖的风险，以及建议如何补充）
```

如果是不变量违反，额外输出：

```text
违反的不变量：
违反位置：
期望状态：
实际状态：
```
