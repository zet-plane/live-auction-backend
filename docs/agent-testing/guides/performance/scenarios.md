# 性能压测场景矩阵

本指南沉淀 `live-auction-backend` 的推荐性能压测场景。具体执行仍必须回到目标模块或流程契约确认业务规则、测试数据、状态不变量和通过标准。

## 出价模块

| 优先级 | 场景 | 核心观测 |
| --- | --- | --- |
| P0 | 出价接口阶梯加压 | QPS、P95/P99、错误率、Redis Lua 耗时、MySQL BidLog 写入耗时 |
| P0 | 热门拍品高并发出价 | 热 key 延迟、锁/原子脚本耗时、价格更新失败率、业务失败码分布 |
| P0 | 出价 + 排行榜查询混合压测 | 写入延迟、查询延迟、Redis ranking 访问耗时、慢查询 |
| P1 | 一口价附近峰值压测 | 成交接口延迟、重复成交错误率、成交后无效出价比例 |
| P1 | 长时间稳定性压测 | 错误率趋势、内存、GC、goroutine、连接池、Redis/MySQL 连接数 |

出价压测后至少抽样验证：

- 最高价、最高出价人、`bid_count`。
- Redis state、ranking 和 MySQL BidLog 是否一致。
- 业务错误码是否符合预期，例如低价失败、成交后出价失败。

## WebSocket 模块

| 优先级 | 场景 | 核心观测 |
| --- | --- | --- |
| P0 | 房间连接数阶梯增加 | 连接成功率、连接耗时、内存、goroutine、FD/连接数 |
| P0 | 出价事件广播扇出 | 端到端推送 P95/P99、丢消息数、慢连接影响 |
| P1 | 心跳和断线重连压力 | 心跳延迟、重连成功率、在线人数 Redis 状态 |

WebSocket 压测后至少抽样验证：

- 房间状态、在线人数、连接索引。
- 关键事件是否被推送，推送延迟是否达标。
- 事件 payload 与 HTTP / Redis / MySQL 最终状态一致。

## 房间和商品模块

| 优先级 | 场景 | 核心观测 |
| --- | --- | --- |
| P1 | 房间列表 / 商品列表查询压测 | QPS、P95/P99、MySQL 慢查询、缓存命中率 |
| P1 | 开播 / 下播 / 拍品开始混合压测 | 状态动作延迟、错误率、DB/Redis 一致性抽样 |

房间和商品压测后至少抽样验证：

- 房间状态和商品状态符合目标契约。
- 列表分页、排序和过滤结果没有异常。
- Redis room queue、item state 和 MySQL 状态一致。

## 场景选择建议

- 首次性能压测优先选择出价接口阶梯加压，先证明 runner、监控、对账和清理闭环可用。
- 需要验证实时体验时，再加入 WebSocket 连接数和广播扇出。
- 需要容量结论时，至少覆盖 smoke、step load 和 peak hold。
- 需要稳定性结论时，在目标或次峰值压力下执行较长时间 soak。
