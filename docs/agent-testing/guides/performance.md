# 性能压测指南

本指南定义 agent 如何设计、执行和报告性能压测。它关注容量、吞吐、延迟、错误率、资源瓶颈和稳定性，不替代并发一致性测试。

通用计划字段、依赖授权、数据隔离、证据、报告、清理、敏感信息和失败输出规则见 `docs/agent-testing/templates/protocol.md`。本文只记录性能压测的附加规则。

并发出价结果唯一性、幂等、状态对账、Redis / MySQL / HTTP 一致性见 `docs/agent-testing/guides/concurrency.md`。

## 适用范围

性能压测用于回答以下问题：

- 目标链路在指定环境中能承载多少 QPS / TPS。
- 目标并发用户数下的 P50、P95、P99 延迟是否达标。
- 错误率、超时率和业务失败率是否在可接受范围内。
- CPU、内存、GC、goroutine、MySQL、Redis、网络或 WebSocket 推送链路中的首个瓶颈在哪里。
- 超过目标压力后，系统是否可控退化，而不是雪崩或产生错误业务状态。

不适合作为性能压测：

- 只验证字段格式、鉴权或普通错误码的接口契约测试。
- 只验证同一状态竞争结果是否唯一的并发一致性测试。
- 使用本地开发机单点发压，试图推导线上容量。
- 没有监控、没有压测批次、没有清理策略的线上直接打压。

## 环境策略

正式性能压测优先使用准生产或线上等价环境。线上压测只能在低风险窗口、具备限流和回滚预案时执行。

压测流量源应与被测服务分离，避免压测机 CPU、网络或连接数先成为瓶颈。需要大流量时使用多台压测机或压测平台，并记录压测源规格。

禁止：

- 对真实用户数据执行破坏性请求。
- 违反 `templates/protocol.md` 的清理、数据隔离和敏感信息规则。
- 在无告警、无限流、无停止开关的情况下直接压线上核心链路。

## 推荐场景矩阵

### 出价模块

| 优先级 | 场景 | 核心观测 |
| --- | --- | --- |
| P0 | 出价接口阶梯加压 | QPS、P95/P99、错误率、Redis Lua 耗时、MySQL BidLog 写入耗时 |
| P0 | 热门拍品高并发出价 | 热 key 延迟、锁/原子脚本耗时、价格更新失败率、业务失败码分布 |
| P0 | 出价 + 排行榜查询混合压测 | 写入延迟、查询延迟、Redis ranking 访问耗时、慢查询 |
| P1 | 一口价附近峰值压测 | 成交接口延迟、重复成交错误率、成交后无效出价比例 |
| P1 | 长时间稳定性压测 | 错误率趋势、内存、GC、goroutine、连接池、Redis/MySQL 连接数 |

### WebSocket 模块

| 优先级 | 场景 | 核心观测 |
| --- | --- | --- |
| P0 | 房间连接数阶梯增加 | 连接成功率、连接耗时、内存、goroutine、FD/连接数 |
| P0 | 出价事件广播扇出 | 端到端推送 P95/P99、丢消息数、慢连接影响 |
| P1 | 心跳和断线重连压力 | 心跳延迟、重连成功率、在线人数 Redis 状态 |

| 优先级 | 场景 | 核心观测 |
| --- | --- | --- |
| P1 | 房间列表 / 商品列表查询压测 | QPS、P95/P99、MySQL 慢查询、缓存命中率 |
| P1 | 开播 / 下播 / 拍品开始混合压测 | 状态动作延迟、错误率、DB/Redis 一致性抽样 |

## 执行步骤

### 1. 定义目标和阈值

每次性能压测必须先写清楚：

- 目标模块、接口或流程。
- 压测环境和被测服务部署规格。
- 压测源数量、规格和网络位置。
- 目标 QPS / TPS 或并发用户数。
- 通过阈值，例如 `P95 < 200ms`、`P99 < 500ms`、错误率 `< 0.1%`。
- 压测时长，例如阶梯加压每档 5 分钟、稳定性压测 30 分钟或更长。
- 停止条件，例如错误率连续 2 分钟超过 5%、数据库 CPU 超过 90%、Redis 超时明显上升。

### 2. 准备批次和数据

每次压测创建独立批次：

```text
batch_id = agent_<target>_load_<YYYYMMDDHHMMSS>
```

准备数据必须满足：

- 用户、商家、房间、拍品等实体可追踪到当前 `batch_id`。
- 压测数据规模能支撑目标流量，不因测试数据太少产生非预期热点。
- 热点场景和非热点场景要明确区分。
- 所有认证 token 只在 runner 或压测工具内使用，不写入报告。

### 3. 记录压测前基线

发起压测前必须记录：

- 被测服务版本、commit、配置摘要。
- 服务实例数、CPU、内存限制。
- MySQL、Redis、网关、负载均衡等依赖规格。
- 目标 HTTP 查询响应和关键业务状态。
- 应用、MySQL、Redis、机器资源的空载指标。

### 4. 执行压测

推荐压测模型：

- Smoke：小流量 1 到 3 分钟，验证脚本、数据和监控。
- 阶梯加压：按目标 QPS 或并发数逐档提升，观察拐点。
- 峰值压测：维持目标峰值，验证是否达标。
- 稳定性压测：在目标或次峰值压力下运行较长时间，观察泄漏和趋势。

压测过程中必须记录每一档：

- 开始时间、结束时间、持续时长。
- 请求总数、成功数、失败数、超时数。
- QPS / TPS。
- P50、P95、P99、最大延迟。
- HTTP 状态码分布和业务错误码分布。
- 资源和依赖指标摘要。

### 5. 观测系统指标

应用至少观测：

- CPU、内存、GC、goroutine。
- 请求耗时、错误率、panic 或错误日志。
- 连接池、队列长度、后台 worker 堆积。

MySQL 至少观测：

- CPU、连接数、慢查询、锁等待。
- 目标表写入和查询耗时。
- 事务冲突或死锁。

Redis 至少观测：

- QPS、延迟、连接数、内存。
- 热 key、Lua 脚本耗时、超时或 rejected connections。

WebSocket 至少观测：

- 连接数、连接成功率、重连率。
- 广播端到端延迟、丢消息数、慢连接影响。

### 6. 对账和判定

性能压测不能只看吞吐和延迟，压测后必须抽样验证业务状态。

出价模块至少抽样验证：

- 最高价、最高出价人、`bid_count`。
- Redis state、ranking 和 MySQL BidLog 是否一致。
- 业务错误码是否符合预期，例如低价失败、成交后出价失败。

房间和 WebSocket 模块至少抽样验证：

- 房间状态、在线人数、连接索引。
- 关键事件是否被推送，推送延迟是否达标。

## 通过标准

压测结论必须明确写出：

- 当前环境下的稳定承载能力。
- 达到目标流量时是否满足延迟和错误率阈值。
- 首个瓶颈和证据。
- 超过目标压力后的退化表现。
- 是否发现业务状态错误。
- 下次压测前需要修复或补充观测的事项。

没有监控数据、没有压测源信息、没有业务状态抽样对账的结果，不能作为正式性能压测结论。
