# Agent 测试执行指南

本指南定义 AI agent 如何使用 `docs/agent-testing` 下的测试契约文档来执行测试。

它不定义具体业务规则。具体业务规则必须来自模块文档或流程文档。

## 核心原则

1. 文档定义边界。
2. Agent 定义执行计划。
3. 测试工具和断言给出证据。
4. 缺少业务规则时，必须先问用户，不允许自行猜测。
5. 发现文档外风险时，只能提出建议，不能直接扩大测试范围。
6. 测试结束后沉淀报告，报告必须能支撑复盘和回归测试设计。
7. 本地代码单元测试必须隔离数据库，使用 mock/fake 数据。
8. Agent 接口契约、模块集成、全流程、并发和状态一致性测试允许连接线上数据库，但只能操作本次测试数据。

## 执行入口

Agent 接到测试任务后，必须遵守 `README.md` 的渐进式读取规则：先读入口，再读本执行指南，再按测试目标读取模块或流程契约。

```text
测试单个模块：
读取 docs/agent-testing/README.md
读取 docs/agent-testing/agent-runner-guide.md
读取 docs/agent-testing/modules/<module>.md

测试跨模块流程：
读取 docs/agent-testing/README.md
读取 docs/agent-testing/agent-runner-guide.md
读取 docs/agent-testing/flows/<flow>.md
只按流程文档要求读取关联模块文档

涉及环境准备、连接数据库/Redis、启动服务或创建测试数据：
再读取 docs/agent-testing/environment.md

测试结束需要沉淀报告：
再读取 docs/agent-testing/reports/README.md
```

如果用户没有明确模块或流程，agent 必须先询问用户要测试哪个目标。

## 执行流程

### 1. 读取测试契约

Agent 必须提取以下内容：

- 测试目标。
- 测试边界。
- 禁止事项。
- 业务规则。
- 业务不变量。
- 测试数据准备。
- 需要覆盖的测试类型。
- 状态一致性验证要求。
- 通过标准。
- 失败报告格式。

### 2. 判断文档是否足够

如果出现以下情况，agent 必须停止并询问用户：

- 业务规则缺失。
- 通过标准缺失。
- 关键状态流转没有定义。
- 并发优先级没有定义。
- 模块文档和流程文档冲突。
- 测试依赖不明确，且会影响测试结论。

Agent 不允许通过猜测补齐这些信息。

### 3. 生成测试计划

执行前，agent 必须输出测试计划。

测试计划至少包含：

```text
测试目标：
读取文档：
测试范围：
禁止范围：
测试类型：
测试数据：
依赖策略：
执行步骤：
验证方式：
预计输出：
```

如果测试计划超出文档边界，必须先征得用户确认。

### 4. 选择依赖策略

Agent 必须根据测试类型选择 mock、fake 或真实依赖。

原则：

> Mock 用来隔离不可控依赖，不是用来逃避真实业务风险。

> 执行接口契约测试、模块集成测试或状态一致性测试时，优先使用 Go runner 采集结构化证据，而不是手工构造散点 curl 命令。参见 `go-runner-guide.md`。

数据库原则：

- 本地代码单元测试禁止直接连接数据库、Redis 或外部服务。
- Agent 执行接口契约、模块集成、全流程、并发和状态一致性测试时，允许连接线上数据库或线上等价真实依赖。
- 连接线上数据库时，必须使用可识别测试数据前缀或测试批次 ID，并记录创建、验证和清理结果。
- 任何测试类型都不允许清空数据库、批量修改非测试数据，或把线上数据库地址、账号、密码写入报告。

#### 单元测试

适合验证纯业务逻辑。

推荐策略：

- Mock 外部系统。
- Fake 时间、ID、随机数。
- 不启动 MySQL。
- 不启动 Redis。
- 不连接 WebSocket。
- 不调用真实 HTTP 服务。
- 不读取线上或本地数据库里的既有数据。

适合 mock 或 fake：

- 短信。
- 支付。
- 第三方 AI 服务。
- 外部通知。
- 时间。
- ID 生成器。
- 随机数。

不应该 mock：

- 当前正在测试的业务规则本身。
- 被测函数内部的核心状态判断。

#### 接口契约测试

适合验证 HTTP 请求、响应、错误码和字段结构。

推荐策略：

- 使用真实 handler 或本地服务。
- Agent 执行时可以使用线上数据库或线上等价真实数据库。
- Agent 执行时可以使用线上 Redis 或线上等价真实 Redis，如果模块依赖 Redis。
- 不 mock response。
- 外部第三方服务仍然 mock。
- 只能创建或修改本次测试数据。

Apifox 对齐步骤：

执行接口契约测试前，agent 应对比目标接口的 Request/Response 字段结构与 Apifox 当前规范是否一致。如有字段偏差，先记录为"Apifox 对齐问题"，再继续执行测试。测试报告中必须单独列出对齐偏差，不得将其混入失败项。

#### 模块集成测试

适合验证模块内部的真实协作。

推荐策略：

- 使用线上数据库或线上等价真实数据库。
- 使用线上 Redis 或线上等价真实 Redis，如果模块依赖 Redis。
- 使用真实 DAO / Service。
- Mock 真实第三方服务。
- 不 mock 当前模块的核心依赖。
- 只能创建或修改本次测试数据。

对出价模块，不应 mock：

- 出价状态计算。
- 当前价更新。
- 出价记录写入。
- Redis 中的实时竞拍状态。
- MySQL 中的出价记录。

#### 全流程测试

适合验证跨模块业务闭环。

推荐策略：

- 使用真实本地服务。
- 使用线上数据库或线上等价真实数据库。
- 使用线上 Redis 或线上等价真实 Redis。
- 使用真实 HTTP 调用。
- WebSocket 如果实现了，应使用真实连接。
- 只 mock 外部不可控服务。
- 只能创建或修改本次测试数据。

对竞拍全生命周期，不应 mock：

- 场次状态流转。
- 出价。
- 排名。
- 落锤。
- 成交结果。
- Redis / MySQL 状态一致性。

#### 并发测试

适合验证竞态、锁、事务和状态一致性。

推荐策略：

- 使用线上数据库或线上等价真实数据库。
- 使用线上 Redis 或线上等价真实 Redis。
- 使用真实并发请求。
- 不 mock 锁机制。
- 不 mock 出价写入。
- 不 mock 场次状态。
- 只能创建或修改本次测试数据。

如果 mock 掉 Redis、DB 或锁机制，并发测试结论通常无效。

并发测试完成后，agent 必须额外输出隔离证明：

```text
并发隔离证明：
- 并发请求实际触发时间窗口（毫秒）：
- 验证最终状态使用的查询方式（SQL / Redis 命令）：
- 并发冲突次数（如有计数机制）：
- 最终状态唯一性证据：
```

#### WebSocket 测试

适合验证实时消息。

推荐策略：

- 使用真实 WebSocket 连接。
- 使用真实业务触发消息。
- 不直接伪造服务端推送结果。
- 最终以查询接口或存储状态校验推送内容。

如果当前实现未支持 WebSocket，按文档标记为不适用并跳过。

### 5. 输出依赖策略表

执行前，agent 必须输出依赖策略表。

示例：

| 依赖 | 使用方式 | 原因 |
| --- | --- | --- |
| MySQL | 线上数据库或线上等价真实库；仅操作本次测试数据 | 需要验证真实持久化状态 |
| Redis | 线上 Redis 或线上等价真实实例；仅操作本次测试 key | 需要验证实时状态和并发一致性 |
| WebSocket | 真实连接 | 需要验证推送内容 |
| 短信 | mock | 不属于竞拍核心链路 |
| 时间 | fake clock | 需要稳定验证开始和结束边界 |

### 6. 执行测试

Agent 执行测试时必须逐项记录：

- 当前测试场景。
- 使用的测试数据。
- 执行的命令或请求。
- 关键响应。
- 数据库或缓存验证结果。
- 是否符合通过标准。

不要只输出“测试通过”或“测试失败”。

### 7. 沉淀测试报告

测试结束后，agent 应读取 `docs/agent-testing/reports/README.md`，并按其中规则生成测试报告。

报告路径：

```text
docs/agent-testing/reports/YYYYMMDD-HHMMSS-<target>-<type>.md
```

报告必须记录：

- 读取了哪些测试契约文档。
- 使用了哪些依赖。
- 执行了哪些步骤。
- 每个结论对应的证据。
- 通过项、失败项和跳过项。
- 测试数据清理结果。
- 建议沉淀的回归测试。

报告不得写入密码、token、生产环境地址或真实用户敏感信息。

### 8. 失败分析

测试失败时，agent 必须按目标文档中的失败报告格式输出。

如果目标文档没有定义失败报告格式，使用以下格式：

```text
失败场景：
复现步骤：
期望结果：
实际结果：
相关证据：
可能原因：
影响范围：
建议修复点：
建议新增的回归测试：
```

### 9. 回归沉淀

发现 bug 或高风险行为后，agent 必须建议沉淀方式。

可选类型：

- 单元测试。
- 接口契约测试。
- 模块集成测试。
- 全流程测试。
- 并发测试。
- 文档化回归场景。

选择标准：

```text
纯业务规则错误 -> 单元测试
HTTP 响应不符合约定 -> 接口契约测试
模块真实依赖协作错误 -> 模块集成测试
跨模块链路错误 -> 全流程测试
竞态或状态覆盖错误 -> 并发测试
业务规则不清晰 -> 补充测试契约文档
```



## Agent 自检清单

执行前检查：

- 是否读取了 `docs/agent-testing/README.md`？
- 是否读取了目标模块或流程文档？
- 是否通过了测试就绪检查（单元测试通过、目标测试所需数据库/Redis 可用）？
- 如果是本地单元测试，是否确认没有直连数据库？
- 如果是 Agent 连接线上数据库，是否限定为本次测试数据并准备记录清理结果？
- 是否明确了测试边界？
- 是否明确了禁止事项？
- 是否明确了核心验证点和辅助验证点？
- 是否输出了依赖策略表？
- 是否存在必须先问用户的问题？

执行后检查：

- 是否每个结论都有证据？
- 是否说明了跳过范围？
- 是否区分了核心验证点和辅助验证点的结论？
- 是否按失败报告格式输出失败（不变量违反时是否额外输出了违反位置和状态）？
- 是否提出了回归沉淀建议？
- 是否按 `docs/agent-testing/reports/README.md` 沉淀报告？
- 是否填写了已知缺口？
- 是否发现需要补充测试契约文档的地方？
