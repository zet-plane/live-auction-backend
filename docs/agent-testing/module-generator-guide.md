# 模块测试文档生成器指南

本指南用于指导 agent 根据代码和已有测试契约，生成或补充 `docs/agent-testing/modules/<module>.md`。

它解决的问题是：当用户说“测试某个模块”时，agent 先生成边界清晰的模块测试文档，再按文档执行测试，而不是直接无边界探索。

## 适用场景

当用户提出以下请求时，agent 应使用本指南：

- 为某个模块生成测试文档。
- 补充某个模块的测试说明。
- 根据代码生成 `docs/agent-testing/modules/<module>.md`。
- 让 agent 后续能按文档测试某个模块。
- 发现测试文档缺失，需要先补测试契约。

## 生成目标

生成器的目标文件固定为：

```text
docs/agent-testing/modules/<module>.md
```

其中 `<module>` 使用小写短横线命名。

示例：

```text
用户模块 -> user.md
拍品模块 -> item.md
出价模块 -> bid.md
竞拍场次模块 -> auction-session.md
规则模块 -> rule.md
```

## 输入来源

生成模块测试文档时，agent 必须遵守渐进式读取规则。先读取入口和本指南，再按需读取模板、相邻模块和目标模块代码。

必读：

1. `docs/agent-testing/README.md`
2. `docs/agent-testing/module-generator-guide.md`
3. `docs/agent-testing/template.md`
4. 目标模块代码

按需读取：

- 已有相邻模块文档，例如 `bid.md`、`auction-session.md`，用于对齐风格和粒度。
- `agent-runner-guide.md`，仅当生成内容需要引用执行流程或依赖策略细节。
- `environment.md`，仅当生成内容涉及环境准备、连接数据库/Redis、启动服务或测试数据清理。

目标模块代码通常包括：

- `internal/app/<module>/router`
- `internal/app/<module>/handler`
- `internal/app/<module>/service`
- `internal/app/<module>/dao`
- `internal/app/<module>/dto`
- `internal/app/<module>/model`

如果模块目录不存在，agent 必须先询问用户模块对应的代码位置。

## 生成流程

### 1. 判断目标模块

Agent 必须先确认：

```text
模块名称：
目标文档路径：
代码目录：
是否已有模块文档：
```

如果用户只说“测试这个模块”，但没有给出模块名，agent 必须先询问。

### 2. 收集代码事实

Agent 需要从代码中提取事实，而不是凭空写业务。

建议收集：

- 路由路径。
- HTTP 方法。
- Handler 名称。
- Request DTO。
- Response DTO。
- Service 方法。
- DAO 方法。
- Model 字段。
- 错误码或错误返回。
- 数据库表。
- Redis key 或缓存访问。
- 外部依赖。
- 日志关键字。

这些信息可以写入测试边界、接口契约测试、状态一致性测试和依赖策略建议。

### 3. 区分事实、推断和待确认

生成文档时必须区分三类内容：

```text
事实：
从代码、已有文档或接口定义中直接得到。

推断：
根据命名、调用关系或已有模块模式得出，但没有明确业务文字。

需确认：
无法从代码判断，且会影响测试边界或通过标准。
```

规则：

- 事实可以直接写入文档。
- 推断可以写入文档，但必须标记为“根据代码结构推断”。
- 需确认内容必须放到“需用户确认的问题”小节，不能伪装成业务规则。

### 4. 生成文档结构

生成的模块文档必须使用 `template.md` 的结构。

额外建议加入以下小节：

```markdown
## 需用户确认的问题
```

当模块文档存在未定义业务规则时，必须列出问题。

示例：

```markdown
## 需用户确认的问题

- 当前模块是否允许重复提交？
- 删除操作是物理删除还是软删除？
- Redis 状态和 MySQL 状态不一致时，以谁为准？
```

如果没有需确认问题，写：

```text
暂无。
```

### 5. 生成依赖策略建议

文档必须说明不同测试类型推荐使用 mock、fake 还是真实依赖。

推荐格式：

```markdown
## 依赖策略建议

| 测试类型 | 依赖策略 | 原因 |
| --- | --- | --- |
| 本地单元测试 | mock/fake 外部依赖、存储、时间和 ID；禁止直连数据库 | 保持业务规则测试稳定，避免依赖外部状态 |
| Agent 接口契约测试 | 使用真实 handler 或本地服务；允许连接线上数据库；mock 外部第三方 | 验证真实请求、响应结构和线上真实状态 |
| Agent 模块集成测试 | 使用真实 DAO / Service；允许连接线上数据库和 Redis | 验证模块真实协作 |
| Agent 并发测试 | 使用线上数据库、Redis 和真实锁机制 | mock 会让并发结论失效 |
```

原则：

- 本地单元测试必须 mock/fake 数据库、缓存和外部依赖。
- Agent 接口契约、模块集成、全流程、并发和状态一致性测试允许连接线上数据库或线上等价真实依赖。
- Agent 连接线上数据库时，只能创建或修改本次测试数据，必须记录测试批次 ID、验证证据和清理结果。
- 并发测试必须使用真实并发控制相关依赖。
- 全流程测试使用真实本地服务和真实依赖。
- 短信、支付、第三方 AI 等外部不可控服务应 mock。

### 6. 写入或更新文档

如果目标文档不存在：

- 按模板创建完整文档。
- 写入从代码得到的事实。
- 写入推断内容并标注来源。
- 写入需用户确认的问题。

如果目标文档已存在：

- 不删除用户已有内容。
- 只补充缺失章节或明显缺失的事实。
- 如果已有内容与代码事实冲突，记录为冲突，不直接覆盖。
- 冲突必须放到“需用户确认的问题”。

### 7. 生成后自检

生成后，agent 必须检查：

- 是否包含模块目标。
- 是否包含测试边界。
- 是否包含禁止事项。
- 是否包含业务规则。
- 是否包含业务不变量。
- 是否包含测试数据准备。
- 是否包含依赖策略建议。
- 是否覆盖单元、接口、业务场景、异常、边界、并发、状态一致性、WebSocket、回归测试。
- 是否包含通过标准。
- 是否包含失败报告格式。
- 是否列出需用户确认的问题。

## 禁止行为

Agent 生成模块测试文档时禁止：

- 编造业务规则。
- 删除用户已有测试契约。
- 将无法确认的推断写成确定事实。
- 因为代码没有实现就删除对应测试风险。
- 将第三方真实服务加入测试依赖。
- 在没有用户确认的情况下扩大到其他模块。
- 生成文档后直接开始测试，除非用户明确要求继续执行。

## 生成器输出格式

生成或更新完成后，agent 必须输出：

```text
目标模块：
目标文档：
读取来源：
新增内容：
保留内容：
推断内容：
需用户确认的问题：
下一步建议：
```
