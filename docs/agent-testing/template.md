# 模块测试文档模板

> 使用方式：复制本模板到 `docs/agent-testing/modules/<module>.md`，然后按模块实际情况填写。不适用的测试类型写明“不适用”和原因。

## 1. 模块目标

说明这个模块负责什么。

## 2. 测试边界

说明 agent 允许测试的范围。

可包含：

- HTTP 接口。
- Service 方法。
- DAO / Model。
- MySQL 表。
- Redis key。
- WebSocket 消息。
- 日志关键字。

## 3. 禁止事项

说明 agent 不允许执行的动作。

示例：

- 不允许清空数据库。
- 不允许修改生产配置。
- 不允许调用真实第三方服务。
- 不允许自行创造业务规则。
- 不允许扩大到当前模块以外的业务。
- 本地单元测试不允许直接连接数据库，必须使用 mock/fake 数据。
- Agent 连接线上数据库时，只能操作本次测试创建的数据或带测试批次 ID 的数据。

## 4. 业务规则

用自然语言描述模块的核心业务规则。

## 5. 业务不变量

列出系统永远不能违反的规则。

示例：

- 已结束的竞拍不能继续出价。
- 成交结果一旦生成，不允许被后续出价改变。

不变量失败时，agent 除常规失败报告外，必须额外输出：

```text
违反的不变量：<不变量名称>
违反位置：<模块/接口/步骤编号>
期望状态：
实际状态：
```

## 6. 测试数据准备

说明测试前需要准备哪些数据。

## 7. 依赖策略建议

说明不同测试类型推荐使用 mock、fake 还是真实依赖。

示例：

| 测试类型 | 依赖策略 | 原因 |
| --- | --- | --- |
| 本地单元测试 | mock/fake 外部依赖、存储、时间和 ID；禁止直连数据库 | 保持业务规则测试稳定，避免依赖外部状态 |
| Agent 接口契约测试 | 使用真实 handler 或本地服务；允许连接线上数据库；mock 外部第三方 | 验证真实请求、响应结构和线上真实状态 |
| Agent 模块集成测试 | 使用真实 DAO / Service；允许连接线上数据库和 Redis | 验证模块真实协作 |
| Agent 并发测试 | 使用线上数据库、Redis 和真实锁机制 | mock 会让并发结论失效 |

## 8. 单元测试

说明哪些纯逻辑需要单元测试。

不适用时说明原因。

## 9. 接口契约测试

说明哪些接口需要验证请求、响应、错误码和字段结构。

不适用时说明原因。

## 10. 业务场景测试

说明正常业务链路如何测试。

建议使用 Given / When / Then 描述。

## 11. 异常测试

说明非法参数、权限不足、状态不允许等情况如何测试。

## 12. 边界测试

说明时间、价格、数量、状态边界如何测试。

## 13. 并发测试

说明是否需要并发测试，以及如何验证并发结果。

不适用时说明原因。

## 14. 状态一致性测试

说明 HTTP 返回、MySQL、Redis、WebSocket、日志之间如何互相验证。

## 15. WebSocket 测试

说明是否需要验证实时消息。

不适用时说明原因。

## 16. 回归测试

说明历史问题或高风险问题应该如何沉淀为回归测试。

## 17. 通过标准

说明什么叫测试通过。

通过标准分两层：

**核心验证点（全部通过才算过）：**

- 说明哪些业务不变量必须成立。
- 说明哪些接口响应必须符合预期。
- 说明哪些存储状态必须一致。

**辅助验证点（建议验证，可附说明跳过）：**

- 说明哪些日志格式、WebSocket 消息等辅助验证点。

每条通过标准必须包含可验证证据。

## 18. 需用户确认的问题

列出无法从代码或已有文档确认，但会影响测试边界或通过标准的问题。

如果没有需确认问题，写：

```text
暂无。
```

## 19. 失败报告格式

测试失败时，agent 必须输出：

```text
失败场景：
复现步骤：
期望结果：
实际结果：
相关证据：
可能原因：
建议修复点：
建议新增的回归测试：
已知缺口：（本次测试因文档或实现原因未覆盖的风险，以及建议如何补充）
```

如果是不变量违反，额外输出：

```text
违反的不变量：
违反位置：
期望状态：
实际状态：
```
