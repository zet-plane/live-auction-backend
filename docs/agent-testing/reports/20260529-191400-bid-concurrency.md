## 执行步骤

1. 执行 `rtk env GOCACHE=/tmp/live-auction-go-cache go test ./...`，确认仓库编译和单元测试通过。
2. 启动本地后端服务：`rtk env GOCACHE=/tmp/live-auction-go-cache go run main.go server -c config.yaml`。
3. 生成并执行 Go runner：`/private/tmp/agent-runner-bid-concurrency/main.go`。
4. Runner 通过真实 HTTP 注册用户、设置 merchant 身份、创建拍品、发布拍品、开始竞拍。
5. 对每个并发场景使用同步起跑 goroutine，记录每个请求的开始/结束时间、HTTP 状态、业务码、`bid_id`、当前价和领先用户。
6. 每个场景结束后查询 Redis state/ranking、MySQL BidLog 聚合和 HTTP ranking 进行对账。
7. Runner 清理本批次 Redis keys、BidLog、auction_rules、auction_items、orders，并软删除本批次用户。
8. 停止临时后端服务。

## 并发设计

| 场景名称 | 竞争对象 | 并发请求 | 预期成功 | 预期失败 | 最终不变量 |
| --- | --- | --- | --- | --- | --- |
| 多用户不同价格同时出价 | Redis current_price / leader / ranking | 8 个不同用户、价格 1100-1800 | 一个或多个合法递增请求 | 被更高并发请求超越后返回 `40003` | 最高成功价在 Redis、DB、HTTP ranking 一致 |
| 多用户相同价格同时出价 | Redis current_price | 8 个不同用户、同价 1100 | 至多 1 个请求改变当前价 | 其余返回 `40003` | BidLog 只有 1 条，ranking 只有 1 个 top |
| 同一幂等 key 重复提交 | Redis idempotency key / BidLog | 同一用户、同一 key、同价 1100，8 次 | 全部幂等成功 | 无业务失败 | 只有一个 `bid_id`，BidLog 和 bid_count 只增加一次 |
| 一口价和普通出价竞争 | MySQL ended 状态 / winner / deal_price | 价格 1400、1500、1600、1700，cap=1500 | 至少一个达到 cap 的请求成交 | 被更高价抢先后返回 `40003` | 成交结果唯一，winner/deal_price 对应成功请求 |
| 排行榜查询与出价并发 | Redis ranking / HTTP ranking | 6 个出价 + 6 个查询 | 查询可见某一时刻快照 | 无乱序或重复 | 最终 Redis、DB、HTTP top 一致 |

## 通过项

- 多用户不同价格同时出价最终最高成功价获胜，Redis state、Redis ranking、MySQL BidLog、HTTP ranking 一致。
- 多用户相同价格同时出价只产生一个有效状态变更，其余请求返回可解释业务错误 `40003`。
- 同一用户同一幂等 key 并发重试只产生一个 `bid_id` 和一条 BidLog，`bid_count` 只增加一次。
- 一口价竞争下成交结果唯一，MySQL `auction_items` 最终为 `ended`，winner/deal_price 与成功请求一致。
- 排行榜查询与并发出价同时发生时，查询响应没有乱序或重复，最终状态对账一致。
- Runner 清理仅限本批次数据，未执行 `TRUNCATE`、`FLUSHDB`、`FLUSHALL` 等危险操作。

## 跳过项

- 同一用户不同幂等 key 快速递增出价：本次已覆盖同一用户同一 key 幂等和多用户价格竞争，但未单独覆盖同一用户多 key 快速递增。影响风险：同一用户 ranking 最高价保留语义未在并发态单独证明。后续补测条件：增加同一用户价格 1100/1200/1300 的同步起跑场景，验证 Redis `ZADD GT` 和 BidLog count。
- WebSocket 推送并发验证：本次聚焦 HTTP/Redis/MySQL 状态一致性。影响风险：客户端实时消息顺序和 payload 未证明。后续补测条件：接入真实 WebSocket 客户端或 fake broadcaster 证据。
- Redis 成功但 MySQL 失败故障注入：本次未模拟持久化失败。影响风险：已知的 Redis/MySQL 短暂不一致补偿语义未证明。后续补测条件：增加故障注入或可控 DAO/DB 错误测试。

## 风险和建议

- 并发不同价格场景中，低价请求在高价先成功后返回 `40003` 是符合当前 Redis Lua 语义的；报告中将其视为可解释业务失败。
- 一口价竞争场景只观察到一个成功请求，未触发多个达到 cap 请求同时成功后的 MySQL ended 条件竞争；建议后续提高并发规模并重复运行该场景。
- 当前实现同步写 BidLog；高并发下如果 MySQL 写入失败，Redis 已更新风险仍需故障注入覆盖。
