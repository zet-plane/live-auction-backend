## 通过项

- `POST /api/v1/ws-ticket` 鉴权、统一错误响应、ticket 格式、TTL 和 Redis value 均符合模块契约。
- `GET /ws/v1/rooms/{room_id}` 首次使用有效 ticket 可 101 upgrade，ticket 被 `GETDEL` 消费，重复使用返回 401。
- `ping` 返回 `pong`。
- `leave_room` 后连接移除，Redis `online_users` / `online_count` 最终回到 0。
- 非法 JSON 和未知客户端事件不会主动关闭连接，后续 ping 仍可收到 pong。
- 报告与 runner 输出均未记录完整 token、完整 ticket 或完整 WebSocket query。

## 失败项

- 无。

## 跳过项

- Hub `Fanout` / `Unicast` 黑盒集成未测：本次没有公开测试入口可直接调用进程内 Hub；对应行为由 `internal/app/ws/hub` 单元测试覆盖。
- 慢连接剔除未做真实 WebSocket 黑盒验证：需要专门构造不读消息的慢客户端并观察 Hub 剔除，建议另立回归或性能/稳定性场景。
- 业务事件 `auction_started`、`bid_success`、`time_sync`、`auction_extended`、`auction_ended`、`order_created` 未在本模块报告中覆盖：这些需要 item/bid/order 真实业务链路，本次只验证 WebSocket 模块入口和在线状态。

## Apifox 对齐偏差

- `/api/v1/ws-ticket` 和 `/ws/v1/rooms/{room_id}` 路径存在，鉴权和 101/400/401 响应与模块文档一致。
- `data.ticket` schema 未声明 32 位 hex pattern，示例为 `ws_ticket`，与实际 32 位 hex ticket 不一致。
- WebSocket 业务事件示例缺少当前 DTO 中新增的 `server_time_unix_ms`、`end_time_unix_ms`、`auction_version` 等字段；本次未执行业务事件链路，因此只记录为规范偏差。

## 风险和建议

- 建议把 ticket schema 更新为 32 位 hex pattern，并补充当前实时事件 payload 字段。
- 建议新增一个可控测试入口或 in-process 集成 harness，用于验证真实 Hub Fanout/Unicast 和慢连接剔除。

## 建议沉淀的回归测试

- 将 ticket 签发、一次性消费、ping/pong、非法消息不关闭连接沉淀为轻量 WebSocket 集成 smoke。
- 将 online state 收敛验证沉淀为 Redis-backed 集成测试。

## 已知缺口

- 本报告不证明业务事件 payload 与 item/bid/order 最终状态一致。
- 本报告不评估 WebSocket 容量、延迟或长期稳定性。
