## 失败项

- WebSocket 握手访问日志泄露完整 ticket query string。

### 失败场景：WebSocket ticket 出现在访问日志中

复现步骤：

1. 启动本地服务。
2. 通过 `/api/v1/ws-ticket` 获取短期 ticket。
3. 使用 `GET /ws/v1/rooms/{room_id}?ticket=<ticket>` 建立 WebSocket 连接。
4. 观察服务 stdout 访问日志。

期望结果：

- 日志不记录完整 query string，或至少对 `ticket` 参数脱敏。

实际结果：

- 访问日志记录了完整 WebSocket URL query，其中包含可复用窗口内的完整 ticket。

相关证据：

- 服务 stdout 中出现 `GET /ws/v1/rooms/<batch-room>?ticket=<full-ticket>` 形状日志；具体 ticket 值已从本报告省略。

可能原因：

- HTTP 访问日志中间件按原始请求 URI 记录路径，没有对 WebSocket ticket query 参数做脱敏。

影响范围：

- ticket TTL 为 45 秒，窗口较短，但在日志采集、集中存储或异常排查链路中仍可能产生凭据泄漏风险。

建议修复点：

- 在 HTTP 访问日志中间件中对敏感 query 参数脱敏，至少覆盖 `ticket`、`token`、`Authorization` 等认证材料。
- WebSocket 路由可记录 path 和状态码，但不要记录完整 query string。

建议新增的回归测试：

- 构造带 `ticket` query 的请求，验证访问日志输出不包含原始 ticket 值。

## 跳过项

- 真实业务事件 `auction_started`、`bid_success`、`user_outbid`、`auction_extended`、`auction_ended`、`order_created` 的端到端 WebSocket 推送未在本次执行中覆盖；本次只覆盖 ws 模块自身 ticket、连接、心跳、并发和 Redis 在线状态。业务事件应在 item / bid / auction lifecycle 场景中联动执行。
- `CheckOrigin` 安全策略未做攻击性验证；当前实现允许所有 origin，按 `ws.md` 记录为需确认/风险项。
- 日志脱敏已发现失败：服务访问日志会记录完整 WebSocket `ticket` query。报告和 runner 输出未记录完整 token / ticket / query string。

## Apifox 对齐偏差

- 当前 Apifox OpenAPI Spec 未包含 `POST /api/v1/ws-ticket`，但代码和测试契约要求前端先通过该接口换取短期一次性 ticket。
- 当前 Apifox `/ws/v1/rooms/{room_id}` 仍描述 query 参数为 `token=jwt_token`；实际实现和 `docs/5-21.md` 要求使用 `ticket=<ws_ticket>`。
- 当前 Apifox WebSocket 响应以 `200` 描述 Switching Protocols；真实 WebSocket upgrade 成功状态为 `101`。
- 当前 Apifox 描述包含 `join_room`、`notification`、`ranking_updated`；当前 ws 连接代码未处理 `join_room`，本次实现中未看到独立 `ranking_updated` 或 `notification` 事件生产。

## 风险和建议

- 建议同步 Apifox：新增 `/api/v1/ws-ticket`，并将 WebSocket query 参数从 `token` 改为 `ticket`。
- 建议优先修复 WebSocket ticket 日志脱敏，避免短期 ticket 出现在 stdout 或集中日志中。
- 建议确认 `online_count` 的业务口径。当前 Redis HASH 按连接数增减，`online_users` Set 按唯一用户存储；同一用户多端连接时二者会不同。
- 建议确认是否需要校验 `room_id` 存在和访问权限。当前实现只校验 room_id 非空。
- 建议确认生产环境 `CheckOrigin` 白名单策略。
- 建议为 WebSocket handler / Conn 增加本地单元或 in-process 测试，当前已有 Hub 和 wsevent 单测，handler 层仍主要由 runner 覆盖。

## 建议沉淀的回归测试

- 同一 ticket 并发握手：只允许 1 个 `101`，其余 `401`。
- WebSocket 访问日志脱敏：日志不得包含完整 `ticket` query 参数值。
- `ping -> pong` 心跳消息结构。
- `leave_room` 后 Redis `online_count` 和 `online_users` 同步减少。
- 缺少 ticket / 无效 ticket 的 400 / 401 分支。
- Apifox 对齐检查：`/api/v1/ws-ticket` 必须存在，WebSocket query 参数必须为 `ticket`。

## 已知缺口

- 本次没有构造完整拍品和出价链路，因此未验证业务事件 payload 与最终 MySQL / Redis 业务状态的一致性。
- 本次没有覆盖同一用户多连接时 `online_count` 与唯一用户 Set 的差异，因为该口径仍待确认。
- 本次没有做大规模扇出、延迟 P95 或慢连接真实 socket 压测。
