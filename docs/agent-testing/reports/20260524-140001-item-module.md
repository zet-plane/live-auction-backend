## 失败项

- 商家调用 `POST /api/v1/items` 时请求体缺少 `room_id`，实际创建成功并返回 HTTP 200，违反模块文档“`room_id` 是 HTTP 创建/修改请求的必填字段”和接口契约“缺少 `room_id` 返回参数错误”。

## 失败详情

### 失败场景

商家创建商品接口缺少 `room_id` 未被绑定层拒绝。

### 复现步骤

1. 注册测试用户并通过 `PUT /api/v1/users/me` 将身份升级为 `merchant`。
2. 使用该商家 token 调用 `POST /api/v1/items`。
3. 请求体包含 `title` 和完整 `rule`，但不包含 `room_id`。

### 期望结果

- HTTP 400。
- 响应 `code=40001`，`message=invalid request`。
- 数据库不创建商品或规则。

### 实际结果

- HTTP 200。
- 响应 `code=0`，`message=ok`。
- 数据库创建了标题 `missing room`、`room_id=''` 的商品。

### 相关证据

- HTTP 摘要：`create_missing_room_with_merchant status=200 code=0 message=ok`。
- 清理证据：对该失败场景产生的单条无前缀测试商品执行定点软删除，`cleaned_unprefixed_missing_room_rows=1`。

### 可能原因

- `internal/app/item/router/item.go` 使用 `binding.JSON(dto.CreateItemRequest{})`，但当前运行时没有对 `binding:"required,min=1,max=64"` 产生的错误进行预期拦截，或 Flamego binding 标签/验证器未按当前 tag 生效。
- `internal/app/item/handler/item.go` 已调用 `web.BindingErrors`，但 `errs` 在缺少 `room_id` 时为空。
- Service 层会保存 `room_id`，但没有独立校验空 `room_id`，因此绑定层漏检后业务层继续创建成功。

### 影响范围

- 商品可在没有直播间归属的情况下创建，破坏“商品必须绑定 room_id”的业务不变量。
- 后续上架、房间待拍队列和状态一致性可能无法正确关联直播间。

### 建议修复点

- 在 `CreateItem`/`UpdateItem` 的 Service 输入校验中补充 `strings.TrimSpace(input.RoomID) != ""`，不要只依赖 HTTP binding。
- 同步检查 Flamego binding tag 是否符合当前版本验证语法，补充 handler 或路由层回归测试。

### 建议新增的回归测试

- HTTP handler/接口测试：商家 `POST /api/v1/items` 缺少 `room_id` 返回 HTTP 400，且数据库无新增商品。
- Service 单元测试：`CreateItem` 传入空白 `RoomID` 返回 `ErrInvalidRequest`。
- Update 测试：修改 draft 商品时缺少或空白 `room_id` 返回参数错误，不应把已有 room_id 清空。

### 不变量违反

- 违反的不变量：商品必须保存创建/修改请求中的 `room_id`，且 HTTP 创建请求 `room_id` 必填。
- 违反位置：`POST /api/v1/items` 创建商品。
- 期望状态：请求被拒绝，商品不创建。
- 实际状态：请求成功，创建了 `room_id=''` 的商品。

## 跳过项

- 并发测试：`item.md` 中并发重复上架、开始、取消的严格语义列为需用户确认，且本次重点为模块接口/状态一致性验证。
- WebSocket 测试：商品模块文档标记不直接测试 WebSocket。
- Redis 故障注入：未强制制造 Redis 写入失败、初始化失败或清理失败；当前已由 service fake cache 单元测试覆盖部分失败路径。
- 线上依赖测试：本次使用本地等价真实依赖，没有连接线上数据库或线上 Redis。

## Apifox 对齐偏差

- `POST /api/v1/items` 的 Apifox requestBody `required` 缺少 `room_id`，但代码 `dto.CreateItemRequest` 和模块契约都要求 `room_id` 必填。
- `POST /api/v1/items` 的 Apifox example 中 `rule.start_price=0`，但 schema `minimum=1`，HTTP binding 也要求 `min=1`。
- 受保护的 item 端点在 Apifox path refs 中 `security=[]`，但运行时需要 Bearer token。
- 多个错误响应 example 复用 `message=invalid bid price` 和 `code=40001`，与真实 unauthorized/not found 响应不一致。

## 风险和建议

- 当前 `config.yaml` 缺少 `redis:` 段，直接 `rtk go run main.go server -c config.yaml` 会失败；建议同步本地配置或在 README 中提示使用完整测试配置。
- `config.yaml.example` 不能直接作为 `-c` 输入使用，Viper 报 `Unsupported Config Type "example"`；建议提供 `config.local.yaml.example` 或在启动命令中明确复制为 `.yaml`。
- 仅依赖 HTTP binding 校验业务关键字段有漏检风险，建议 Service 层保留不可变业务字段校验。
- Apifox 鉴权和错误示例偏差会影响接口契约测试预期，应单独同步。
