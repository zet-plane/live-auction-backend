## 并发隔离证明

- 并发请求实际触发时间窗口（毫秒）：0ms；请求起跑时间集中在同一毫秒内
- 验证最终状态使用的查询方式：HTTP `GET /api/v1/items/{item_id}/ranking`、SQL 查询 `bid_logs` 聚合、Redis `HGETALL` state、Redis `ZREVRANGE WITHSCORES` ranking
- 并发冲突次数：3 次低价冲突，均返回 `40003 price too low`
- 最终状态唯一性证据：Redis state leader、Redis ranking 第一名、HTTP ranking 第一名、MySQL 最高价聚合均指向同一用户和价格 1500

## 风险和建议

- 本次结果中只有 1400 和 1500 成功，低价请求因更高价先行写入 Redis 后被拒绝；这符合当前 Lua 的当前价原子校验语义。
- 建议后续补测“多用户相同价格同时出价”和“一口价与普通出价同时发生”，覆盖同价唯一性和成交唯一性。
- 当前 runner 为一次性 `/private/tmp` 工具；建议将该场景转化为可复用的受控 agent runner 或集成测试脚本。

## 建议沉淀的回归测试

- 多用户不同价格同步出价后，最高成功价必须成为 Redis state、Redis ranking、HTTP ranking 和 MySQL 聚合的唯一第一名。
- 并发失败请求不得写入 BidLog、ranking 或 idempotency key。
- 并发请求中低价失败必须返回可解释业务错误码 `40003`。

## 已知缺口

- 未覆盖 Redis 成功但 MySQL 写 BidLog 失败的故障注入路径。
- 未覆盖同价排序规则，因为本场景价格均不同。
- 未覆盖商家身份是否允许出价、报名/风控/黑名单等待确认业务规则。
