# 测试报告沉淀目录

本目录用于沉淀 agent 执行测试后的报告。

报告的目标是让测试过程可以追踪、复盘和转化为回归测试，而不是只停留在一次聊天结论里。

## 读取入口

按渐进式读取规则，只有当 agent 需要写入、补充或校验测试报告时，才读取本文档。

执行测试前不需要预读完整报告模板；只需在测试计划中说明测试完成后会按本文档沉淀报告。

## 命名规则

报告文件使用 Markdown，命名格式：

```text
YYYYMMDD-HHMMSS-<target>-<type>.md
```

示例：

```text
20260523-153000-bid-module.md
20260523-160000-auction-lifecycle-flow.md
20260523-170000-bid-concurrency.md
```

字段说明：

- `YYYYMMDD-HHMMSS`：测试开始时间。
- `target`：测试目标，例如 `bid`、`auction-session`、`auction-lifecycle`。
- `type`：测试类型，例如 `module`、`flow`、`contract`、`concurrency`、`regression`。

## 报告内容

每份报告必须包含：

```text
测试目标：
测试类型：
测试时间：
执行 agent：
读取文档：
测试环境：
依赖策略：
测试数据：
执行步骤：
验证证据：
通过项：
失败项：
跳过项：
Apifox 对齐偏差：（接口契约测试时填写；无偏差写"无"）
风险和建议：
建议沉淀的回归测试：
已知缺口：（本次因文档或实现原因未覆盖的风险）
测试数据清理结果：
```

填写要求：

- `验证证据` 必须把“命令 / 请求 / 数据库或 Redis 证据 / 结果”对应起来，不能只写通过结论。
- `跳过项` 必须说明跳过原因、影响风险和后续补测条件；如果无跳过项，写“无”。
- `Apifox 对齐偏差` 只记录接口规范与代码/模块文档的差异，例如字段缺失、默认值不一致、枚举不一致；不要把业务失败混入这一节。
- 如果 runner 或服务启动时遇到沙箱权限问题，报告只记录最终使用的依赖方式和必要证据，不记录敏感地址或凭据。
- 如果 agent 临时启动了本地服务，报告应说明测试结束后是否已停止服务或释放端口。

## 失败报告内容

如果存在失败项，必须包含：

```text
失败场景：
复现步骤：
期望结果：
实际结果：
相关证据：
可能原因：
影响范围：
建议修复点：
建议新增的回归测试：
```

## 证据要求

报告中的结论必须有证据支撑。

可接受证据包括：

- 测试命令和退出码。
- HTTP 请求和响应摘要。
- WebSocket 消息摘要。
- MySQL 查询结果摘要。
- Redis 查询结果摘要。
- 关键日志片段。
- Go 测试输出摘要。

不要粘贴过长日志。长日志应只记录关键片段、文件路径或检索方式。

## 敏感信息规则

报告中禁止写入：

- 数据库密码。
- Redis 密码。
- Apifox token。
- 生产环境地址。
- 线上数据库地址。
- 线上 Redis 地址。
- 真实用户手机号。
- 真实支付信息。
- 任何可复用的认证 token。

如果必须记录认证相关信息，只能写：

```text
认证方式：使用测试环境 token，具体值已省略。
```

## 测试数据清理

如果测试创建了数据，报告必须记录：

```text
测试批次 ID：
创建的数据：
清理方式：
清理结果：
未清理原因：
```

Agent 不能为了清理测试数据执行危险操作，例如：

- `DROP DATABASE`
- `DROP TABLE`
- `TRUNCATE`
- `FLUSHALL`
- `FLUSHDB`

只能清理本次测试创建的数据，或带有明确测试前缀 / 测试批次 ID 的数据。

如果测试连接了线上数据库或线上 Redis，报告必须记录：

```text
线上依赖使用情况：已使用，地址和凭据已省略。
测试数据范围：仅本次测试批次 ID / 测试前缀。
清理结果：
```
