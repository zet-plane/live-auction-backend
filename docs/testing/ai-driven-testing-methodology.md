# AI 驱动测试方法论

## 背景

当前测试方式主要来自两类：

1. 开发者编写单元测试。
2. 通过 Apifox、curl 或其他工具进行手动接口测试。

这两类方式仍然重要，但在 agent 参与后，测试可以进一步升级为一套由 AI 引导、执行、分析并沉淀的闭环流程。

核心目标不是让 AI 凭感觉判断系统是否正确，而是让 AI 成为测试流程的指挥官：

- 引导开发者澄清业务规则。
- 提炼系统不变量。
- 生成测试场景。
- 调用接口、数据库、缓存、日志和 WebSocket 等工具执行验证。
- 分析失败原因。
- 将问题沉淀为可重复执行的回归测试。

一句话概括：

> AI 负责生成和执行测试思路，人负责定义业务真理，代码负责确定性裁判。

## 基本原则

### AI 不替代断言

AI 可以负责：

- 阅读需求和代码。
- 设计测试策略。
- 编排测试步骤。
- 构造测试数据。
- 调用 HTTP / WebSocket / Apifox / curl。
- 查询 MySQL、Redis 和日志。
- 分析失败现象。
- 生成复现步骤和修复建议。

但最终判断必须由确定性机制完成：

- HTTP 状态码。
- 响应体字段。
- 业务错误码。
- 数据库状态。
- Redis 状态。
- WebSocket 消息内容和顺序。
- Go 测试中的断言。

也就是说，AI 是测试工程师，不是断言引擎。

### 先有业务真理，再有测试执行

不要让 agent 直接开始调用接口。正确顺序是：

1. 明确业务规则。
2. 明确系统不变量。
3. 明确高风险场景。
4. 再生成测试场景。
5. 最后执行测试。

对直播竞拍系统来说，典型不变量包括：

- 未开始的场次不能出价。
- 已结束的场次不能出价。
- 出价必须高于当前价，并满足最小加价规则。
- 当前最高价必须等于最高有效出价。
- 排名第一必须等于当前最高有效出价人。
- 落锤成交后，成交结果不能被后续出价改变。
- 同一拍品不能重复成交。
- Redis、MySQL 和对外查询结果不能长期不一致。

## Agent 引导流程

Agent 不应该只在最后执行测试，而应该从测试设计阶段就开始引导开发者。

整体流程如下：

```text
你提出模块
  ↓
agent 追问业务规则
  ↓
agent 生成不变量
  ↓
agent 生成测试场景
  ↓
你确认是否符合业务
  ↓
agent 编写或执行测试
  ↓
agent 分析失败
  ↓
agent 修复或提出修复建议
  ↓
agent 沉淀回归测试
```

## 五个阶段

### 阶段 1：测试导航员

Agent 先不写测试，而是围绕模块向开发者提问。

目标是把开发者脑中的业务判断显性化。

典型问题：

```text
这个模块的核心业务目标是什么？
哪些行为绝对不能错？
哪些场景出问题代价最大？
当前最担心接口、状态、并发，还是数据一致性？
```

对竞拍系统，agent 应重点追问：

```text
竞拍什么时候开始？
竞拍什么时候结束？
出价成功的条件是什么？
并发出价时谁赢？
落锤后是否还能出价？
Redis 和 MySQL 谁是事实来源？
WebSocket 推送失败时是否影响成交？
```

阶段产物：

- 业务规则清单。
- 风险清单。
- 测试优先级。

### 阶段 2：测试设计师

Agent 根据业务规则，把测试拆成可执行场景。

测试场景建议分为：

- 正常链路。
- 异常链路。
- 边界链路。
- 并发链路。
- 状态恢复链路。
- 权限链路。

示例：

```text
场景：用户参与竞拍并成功出价

Given:
- 拍品已创建
- 竞拍场次已开始
- 当前价为 1000
- 最小加价为 100

When:
- 用户 A 出价 1100

Then:
- 出价成功
- 当前价变为 1100
- 排名第一为用户 A
- WebSocket 推送最新排名
- Redis 当前价与查询接口返回一致
```

阶段产物：

- `docs/testing/invariants.md`
- `docs/testing/scenarios.md`
- 测试优先级矩阵

### 阶段 3：测试执行官

Agent 判断不同场景应该使用哪种测试方式。

推荐映射：

| 测试对象 | 推荐方式 |
| --- | --- |
| 纯业务规则 | 单元测试 |
| HTTP 请求和响应结构 | 接口契约测试 |
| 多接口业务流程 | 集成测试 / 场景测试 |
| 多用户同时出价 | 并发测试 |
| WebSocket 推送 | 端到端消息测试 |
| Redis / MySQL 一致性 | 状态校验测试 |
| 历史 bug | 回归测试 |

Agent 执行测试时应该说明当前验证目标。

示例：

```text
我现在要验证：落锤后不允许继续出价。
我会先创建场次，再结束场次，再尝试出价。
如果接口返回成功，就是业务漏洞。
如果接口返回失败，还要确认数据库没有写入脏出价。
```

阶段产物：

- 可执行测试命令。
- 测试数据准备步骤。
- 接口调用记录。
- 数据库 / 缓存 / 日志验证结果。

### 阶段 4：失败分析师

当测试失败时，agent 不应该只输出“测试失败”，而应该输出可行动的分析。

标准失败报告：

```text
失败场景：

复现步骤：

期望结果：

实际结果：

相关日志：

可能原因：

建议修复点：

需要新增的回归测试：
```

这个阶段的目标是把失败从“现象”推进到“可修复的问题”。

### 阶段 5：回归资产管理员

每次发现问题后，agent 都需要判断是否要沉淀为长期测试资产。

判断问题：

```text
这个问题以后还可能出现吗？
这个问题是否影响核心业务？
这个问题是否容易被重构或新功能再次引入？
这个问题应该沉淀成单元测试、集成测试，还是场景测试？
```

可沉淀资产：

- 单元测试。
- 接口契约测试。
- 业务场景测试。
- 并发测试。
- 回归测试文档。
- Apifox 用例。
- CI 测试任务。

## 测试分层

### 第一层：单元测试

适合测试纯逻辑：

- 加价规则。
- 竞拍状态机。
- 排名计算。
- 时间窗口判断。
- 权限判断。

### 第二层：接口契约测试

适合测试接口是否符合约定：

- 状态码。
- 响应结构。
- 错误码。
- 必填字段。
- 类型约束。

这层可以结合 OpenAPI、Apifox 或路由信息生成。

### 第三层：业务场景测试

适合测试完整业务流程：

```text
创建商家
创建拍品
创建竞拍场次
配置竞拍规则
开始竞拍
用户 A 出价
用户 B 加价
查询排名
触发落锤
查询成交结果
结束后再次出价，应失败
```

### 第四层：探索式和对抗式测试

适合发现边界问题：

- 重复请求。
- 并发出价。
- 时间边界。
- 非法价格。
- 不存在的拍品。
- 已下架拍品。
- Redis 和 MySQL 状态不一致。
- WebSocket 推送丢失或乱序。

## 直播竞拍系统的测试优先级

建议优先级如下：

| 优先级 | 测试方向 |
| --- | --- |
| P0 | 出价正确性 |
| P0 | 竞拍状态流转 |
| P0 | 落锤成交唯一性 |
| P0 | 并发出价一致性 |
| P1 | 排名计算 |
| P1 | WebSocket 推送 |
| P1 | Redis / MySQL 状态一致性 |
| P2 | 后台管理接口 |
| P2 | 异常参数和权限校验 |
| P3 | AI 氛围 / 动态定价相关能力 |

尤其需要重点验证：

```text
多个人同时出价时，谁赢？
落锤瞬间还能不能出价？
Redis 和 MySQL 不一致时以谁为准？
重复请求会不会重复成交？
WebSocket 推送的排名是不是和服务端真实排名一致？
```

## 最小落地版本

建议先从一个轻量版本开始，不要一开始追求完整平台化。

推荐目录：

```text
docs/testing/invariants.md
docs/testing/scenarios.md
docs/testing/ai-agent-guide.md
tests/scenario/auction_flow_test.go
tests/scenario/concurrent_bid_test.go
tests/contract/api_contract_test.go
```

最小闭环：

```text
1. 选择一个模块，例如出价模块。
2. Agent 引导开发者回答业务规则。
3. Agent 生成不变量。
4. Agent 生成 5 到 10 个核心测试场景。
5. 开发者确认业务含义。
6. Agent 编写或执行测试。
7. Agent 输出失败分析。
8. 修复后补充回归测试。
```

## Agent 提示词模板

### 测试导航员模板

```text
你现在是 live-auction-backend 的 AI 测试导航员。

不要直接写测试。
请先围绕当前模块，逐步引导我明确：
1. 核心业务规则
2. 不变量
3. 高风险场景
4. 正常 / 异常 / 边界 / 并发测试场景
5. 哪些测试应该自动化

每次只问我一个问题。
当信息足够时，输出测试设计文档。
```

### 测试执行官模板

```text
你现在是 live-auction-backend 的 AI 测试执行官。

请读取 docs/testing 下的测试说明。
启动本地服务和必要依赖。
准备测试数据。
执行核心竞拍链路。
验证 HTTP、数据库、Redis、WebSocket 和日志。

如果失败，请输出：
1. 失败场景
2. 最小复现步骤
3. 期望结果
4. 实际结果
5. 可能原因
6. 建议新增的回归测试
```

### 回归沉淀模板

```text
你现在是 live-auction-backend 的回归测试管理员。

请分析刚刚发现或修复的问题。
判断它应该沉淀成哪类测试：
- 单元测试
- 接口契约测试
- 业务场景测试
- 并发测试
- 文档化测试用例

然后补充对应测试，并说明这个测试未来能防止什么问题再次出现。
```

## 推荐推进顺序

第一阶段，先做文档化：

```text
docs/testing/invariants.md
docs/testing/scenarios.md
```

第二阶段，做出价模块的场景测试：

```text
用户正常出价
低于最低加价失败
未开始出价失败
已结束出价失败
落锤后出价失败
```

第三阶段，做并发出价测试：

```text
多个用户同时出价
相同价格同时出价
落锤瞬间并发出价
重复请求出价
```

第四阶段，接入 Apifox / OpenAPI：

```text
由 API 规格生成接口契约测试
由 agent 对比实际响应和规格差异
将稳定用例沉淀回 Apifox 或代码测试
```

第五阶段，接入 CI：

```text
每次提交自动执行核心回归测试
失败时由 agent 分析失败原因
生成可读的测试报告
```

## 结论

AI 驱动测试的关键不是“让 AI 多写几个测试”，而是建立一条闭环：

```text
业务规则
  ↓
不变量
  ↓
测试场景
  ↓
自动执行
  ↓
失败分析
  ↓
修复验证
  ↓
回归沉淀
```

对 live-auction-backend 来说，建议从出价模块和竞拍状态模块开始。它们状态复杂、边界多、并发风险高，最能体现 agent 引导测试的价值。
