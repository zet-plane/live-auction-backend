# 直播竞拍系统接口与模型设计 V1

本文档基于「实时竞拍大师」直播竞拍系统需求，整理后端核心模型、HTTP API、WebSocket 事件、状态机以及高并发一致性设计。

## 1. 通用约定

### 1.1 API 前缀

所有 HTTP 接口统一使用：

```text
/api/v1
```

WebSocket 接口统一使用：

```text
/ws/v1
```

### 1.2 认证方式

登录成功后返回 JWT，后续需要登录的接口通过 Header 携带：

```text
Authorization: Bearer <token>
```

### 1.3 统一响应格式

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败响应：

```json
{
  "code": 40001,
  "message": "invalid bid price",
  "data": null
}
```

### 1.4 金额单位

所有金额字段统一使用 `int64`，单位为分，避免 `float` 带来的精度问题。

例如：

- `100` 表示 1 元
- `10000` 表示 100 元

### 1.5 用户角色

```go
type UserIdentity string

const (
    IdentityUser     UserIdentity = "user"
    IdentityMerchant UserIdentity = "merchant"
)
```

## 2. 用户模块

由于用户和商家不是本次系统核心，用户模块只保留基本字段与账号密码注册登录能力。

### 2.1 User Model

```go
type User struct {
    ID        string       `json:"id"`
    Account   string       `json:"account"`
    Name      string       `json:"name"`
    AvatarURL string       `json:"avatar_url"`
    Password  string       `json:"-"`
    Motto     string       `json:"motto"`
    Identity  UserIdentity `json:"identity"`

    DeletedAt *time.Time   `json:"deleted_at,omitempty"`
    CreatedAt time.Time    `json:"created_at"`
    UpdatedAt time.Time    `json:"updated_at"`
}
```

字段说明：

- `Account` 为登录账号，要求全局唯一。
- `Password` 不返回给前端。
- `AvatarURL` 使用字符串存储，方便 JSON 与数据库处理。
- `Identity` 用于区分普通用户和商家。
- 用户可通过修改资料接口升级为商家身份。

## 3. 商品与竞拍规则模块

商品由商家创建。创建商品时可以同步创建竞拍规则。

### 3.1 AuctionItemStatus

```go
type AuctionItemStatus string

const (
    ItemDraft     AuctionItemStatus = "draft"
    ItemPublished AuctionItemStatus = "published"
    ItemOngoing   AuctionItemStatus = "ongoing"
    ItemEnded     AuctionItemStatus = "ended"
    ItemCancelled AuctionItemStatus = "cancelled"
)
```

### 3.2 AuctionItem Model

```go
type AuctionItem struct {
    ID          string            `json:"id"`
    MerchantID  string            `json:"merchant_id"`

    Title       string            `json:"title"`
    Description string            `json:"description"`
    ImageURL    string            `json:"image_url"`
    Tags        []string          `json:"tags"`

    Status      AuctionItemStatus `json:"status"`
    RuleID      string            `json:"rule_id"`

    // CurrentPrice int64            `json:"current_price"` // 从 Redis 中获取
    WinnerID     string           `json:"winner_id,omitempty"`
    DealPrice    int64            `json:"deal_price"`

    CreatedAt    time.Time        `json:"created_at"`
    UpdatedAt    time.Time        `json:"updated_at"`
}
```

字段说明：

- `CurrentPrice` 表示当前最高价。
- `WinnerID` 表示当前领先用户或最终获胜用户。
- `DealPrice` 表示最终成交价。
- `RuleID` 关联竞拍规则。

### 3.3 AuctionRule Model

```go
type AuctionRule struct {
    ID                string    `json:"id"`
    ItemID            string    `json:"item_id"`

    StartPrice        int64     `json:"start_price"`
    BidIncrement      int64     `json:"bid_increment"`
    PriceCap          int64     `json:"price_cap"`
    DepositAmount     int64     `json:"deposit_amount"`

    StartTime         time.Time `json:"start_time"`
    EndTime           time.Time `json:"end_time"`

    CreatedAt         time.Time `json:"created_at"`
    UpdatedAt         time.Time `json:"updated_at"`
}
```

字段说明：

- `StartPrice` 支持 0 元起拍。
- `BidIncrement` 表示每次固定加价幅度。
- `PriceCap` 表示封顶价，达到后自动成交。
- `DepositAmount` 表示保证金金额。
- 自动延时策略不由商家自由配置，使用平台级竞拍策略控制。

### 3.4 AuctionPolicy 平台竞拍策略

自动延时规则会影响竞拍公平性和系统稳定性，因此作为平台级策略统一配置，不开放给商家任意填写。

```go
type AuctionPolicy struct {
    ExtendTriggerSec  int `json:"extend_trigger_sec"`
    AutoExtendSec     int `json:"auto_extend_sec"`
    MaxExtendCount    int `json:"max_extend_count"`
    MaxTotalExtendSec int `json:"max_total_extend_sec"`
}
```

字段说明：

- `ExtendTriggerSec`：距离结束多少秒内出价会触发延时，例如最后 30 秒内有人出价。
- `AutoExtendSec`：每次自动延长多少秒，例如每次延长 10 秒。
- `MaxExtendCount`：最多自动延长次数，防止竞拍被无限拖长。
- `MaxTotalExtendSec`：最多累计延长时间，防止单场竞拍持续过久。

示例配置：

```yaml
auction:
  extend_trigger_sec: 30
  auto_extend_sec: 10
  max_extend_count: 6
  max_total_extend_sec: 300
```

```json

  "description": "天然翡翠，支持鉴定",
  "image_url": "https://example.com/item.png",

    "deposit_amount": 5000,
```

```json

  "code": 0,
  "message": "ok",
  "data": {
    "item_id": "item_123",
```

容量与时效性要求：

- 数据时效性要求：强一致。
- 说明：创建商品和规则需要在 MySQL 中事务写入，避免商品已创建但规则缺失。

### 3.12 开始竞拍

POST /api/v1/items/{item_id}/start

该接口用于商家手动立即开始竞拍，也可作为定时任务内部复用的启动逻辑。正常情况下，商品上架后会根据规则中的 `start_time` 定时自动开始。

状态流转：

published -> ongoing

开始竞拍时需要：

- 校验商品状态必须是 `published`
- 使用数据库条件更新或分布式锁防止重复开始
- 初始化 Redis 中的竞拍状态、当前价格、排行榜、结束时间
- 广播 `auction_started` WebSocket 事件

定时开始说明：

- 商品上架后，将 `item_id` 和 `start_time` 写入定时任务
- 到达 `start_time` 后由后台 worker 自动触发开始竞拍
- 前端列表和详情页展示开拍倒计时
- 客户端时间不参与裁决，以服务端时间为准

容量与时效性要求：

- 高峰期预估 QPS：单商家小于 2 QPS。
- 响应速率要求：P95 小于 300ms。
- 数据时效性要求：强一致。
- 说明：开始竞拍时需要初始化 Redis 中的竞拍状态、当前价格、排行榜、结束时间，并广播 `auction_started` 事件。接口需要防止重复开始。

### 3.13 取消竞拍

```text
POST /api/v1/items/{item_id}/cancel
```

需要商家身份。

状态流转：

```text
published -> cancelled
ongoing -> cancelled
```

取消后需要广播 WebSocket 事件。

容量与时效性要求：

- 高峰期预估 QPS：单商家小于 2 QPS。
- 响应速率要求：P95 小于 300ms。
- 数据时效性要求：强一致，并要求实时广播。
- 说明：取消后需要更新 MySQL 和 Redis 状态，并通过 WebSocket 通知直播间用户。

## 4. 出价与排行榜模块

出价是系统核心链路，需要保证实时性、一致性和幂等性。

### 4.1 BidLog Model

```go
type BidLog struct {
    ID        string    `json:"id"`
    ItemID    string    `json:"item_id"`
    RoomID    string    `json:"room_id"`
    UserID    string    `json:"user_id"`
    Price     int64     `json:"price"`
    CreatedAt time.Time `json:"created_at"`
}
```

### 4.2 出价接口

```text
POST /api/v1/items/{item_id}/bids
```

需要登录。

请求：

```json
{
  "price": 1200,
  "idempotency_key": "uuid-from-client"
}
```

校验规则：

- 商品状态必须是 `ongoing`。
- 当前时间必须在竞拍时间范围内。
- 出价必须大于当前最高价。
- 出价必须满足固定加价幅度。
- 出价不能低于起拍价。
- 出价达到封顶价时自动成交。
- 结束前指定时间内出价，自动延长结束时间。
- 同一个 `idempotency_key` 只能成功一次，避免重复出价。

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "bid_id": "bid_123",
    "current_price": 1200,
    "leader_user_id": "user_123",
    "end_time": "2026-05-21T20:10:10+08:00",
    "status": "ongoing"
  }
}
```

容量与时效性要求：

- 高峰期预估 QPS：单直播间基础目标按 100-300 QPS 设计，活动峰值可按 500+ QPS 预留。
- 响应速率要求：P95 小于 100ms，P99 小于 200ms。
- 数据时效性要求：强实时。必须基于 Redis 中最新的当前价、竞拍状态和结束时间进行校验，不能读取过期缓存。
- 说明：这是系统最高优先级接口。同一商品的并发出价需要通过 Redis Lua 脚本或分布式锁串行化处理，避免并发覆盖和重复成交。

### 4.4 Redis 实时数据设计

Redis 实时数据按两个维度设计：

- `room` 维度：负责直播间上下文、在线用户、房间事件广播和当前正在竞拍的商品。
- `item` 维度：负责某一件商品的竞拍状态、排行榜、出价记录和幂等控制。

房间维度：

```text
auction:room:{room_id}:state
auction:room:{room_id}:online_users
auction:room:{room_id}:current_item
auction:room:{room_id}:item_queue
auction:room:{room_id}:events
```

商品维度：

```text
auction:item:{item_id}:state
auction:item:{item_id}:ranking
auction:item:{item_id}:bid_logs
auction:item:{item_id}:idempotency:{key}
```

字段说明：

- `auction:room:{room_id}:state`：直播间状态，例如是否直播中、商家 ID、当前在线人数等。
- `auction:room:{room_id}:online_users`：直播间在线用户集合，用于统计在线人数和房间级广播。
- `auction:room:{room_id}:current_item`：当前直播间正在讲解或竞拍的商品 ID。
- `auction:room:{room_id}:item_queue`：直播间商品队列，用于支持一个直播间连续竞拍多件商品。
- `auction:room:{room_id}:events`：房间事件流，例如进入房间、开始竞拍、出价成功、竞拍结束等，可用于断线重连后的事件补偿。
- `auction:item:{item_id}:state`：单件商品的竞拍实时状态，包括当前价、领先用户、结束时间、出价次数等。
- `auction:item:{item_id}:ranking`：单件商品的实时排行榜，使用 Redis Sorted Set，score 为用户最高出价。
- `auction:item:{item_id}:bid_logs`：单件商品的近期出价记录，用于直播间动态展示和异步落库。
- `auction:item:{item_id}:idempotency:{key}`：单次出价请求的幂等记录，防止重复点击或网络重试导致重复出价。

建议 Redis 类型：

- `auction:room:{room_id}:state` 使用 `HASH` 或 `STRING(JSON)`。
- `auction:room:{room_id}:online_users` 使用 `SET`。
- `auction:room:{room_id}:current_item` 使用 `STRING`。
- `auction:room:{room_id}:item_queue` 使用 `LIST` 或 `ZSET`。
- `auction:room:{room_id}:events` 使用 `STREAM` 或 `LIST`。
- `auction:item:{item_id}:state` 使用 `HASH` 或 `STRING(JSON)`。
- `auction:item:{item_id}:ranking` 使用 `ZSET`。
- `auction:item:{item_id}:bid_logs` 使用 `LIST`。
- `auction:item:{item_id}:idempotency:{key}` 使用 `STRING(JSON)`，并设置过期时间。

设计原则：

- 房间负责“谁在这个直播间、当前拍哪件商品、房间内发生了什么”。
- 商品负责“这件商品当前多少钱、谁领先、排行榜是什么、出价是否重复”。
- 出价成功后可以异步或批量落库到 MySQL。
- 竞拍结束时以 Redis 中的最终状态为准结算，再写回 MySQL。
- 用户断线重连后，先通过 `room_id` 找到 `current_item`，再拉取商品详情和排行榜恢复现场。

### 4.5 排行榜 SQL 兜底查询

MySQL 中可通过出价日志计算排行榜，用于历史查询或 Redis 失效后的兜底。

```sql
SELECT
    user_id,
    MAX(price) AS highest_price
FROM bid_logs
WHERE item_id = ?
GROUP BY user_id
ORDER BY highest_price DESC
LIMIT 10;
```

- 响应速率要求：P95 小于 300ms。
- 数据时效性要求：历史通知允许 1-5 秒延迟；关键实时通知优先通过 WebSocket 推送，响应速度应该在 1 秒以内。
- 说明：通知列表用于补偿查询和历史回看，不承担实时竞拍主链路。

### 9.4 标记通知已读

```text
POST /api/v1/notifications/{notification_id}/read
```

需要登录。

容量与时效性要求：

- 高峰期预估 QPS：单用户小于 10 QPS。
- 响应速率要求：P95 小于 300ms。
- 数据时效性要求：允许 1-3 秒延迟。
- 说明：标记已读是低频状态更新，后端需要校验通知归属。

## 10. 核心状态机

### 10.1 商品状态流转

```text
draft -> published -> ongoing -> ended
draft -> cancelled
published -> cancelled
ongoing -> cancelled
published -> draft -> published
```

状态说明：

- `draft`：草稿，商家可编辑。
- `published`：已上架，等待开始竞拍。
- `ongoing`：竞拍中。
- `ended`：竞拍正常结束。
- `cancelled`：竞拍被商家取消。

### 10.2 订单状态流转

```text
pending -> paid
pending -> cancelled
pending -> expired
```

状态说明：

- `pending`：待支付。
- `paid`：已支付。
- `cancelled`：用户取消或系统取消。
- `expired`：超时未支付。

### 10.3 竞拍结束规则

竞拍结束条件：

- 当前时间超过 `end_time`。
- 出价达到 `price_cap`。
- 主播或商家手动取消。

正常结束动作：

- 锁定最终最高价和获胜用户。
- 写入商品成交信息。
- 生成订单。
- 广播 `auction_ended` 事件。
- 持久化 Redis 中的最终状态。

取消动作：

- 商品状态变为 `cancelled`。
- 不生成订单。
- 广播 `auction_cancelled` 事件。

## 11. 高并发一致性设计

### 11.1 出价流程

建议出价流程：

1. 客户端提交出价金额和 `idempotency_key`。
2. 服务端校验 JWT 和用户身份。
3. 使用 Redis 分布式锁锁定当前 `item_id`。
4. 读取 Redis 中的当前竞拍状态。
5. 校验商品状态、竞拍时间、价格、加价幅度。
6. 检查 `idempotency_key` 是否已经使用。
7. 更新当前最高价、领先用户、排行榜、结束时间。
8. 写入出价记录。
9. 广播 WebSocket 事件。
10. 如果达到封顶价或满足结束条件，触发结算。

### 11.2 核心原则

- 高频读写优先走 Redis。
- MySQL 保存最终可信业务数据。
- 出价接口必须保证幂等。
- WebSocket 只负责实时同步，不作为最终数据来源。
- 页面刷新或重连后，应通过 HTTP 商品详情接口恢复状态。
- 竞拍结束时需要做一次最终结算，避免 Redis 和 MySQL 状态不一致。

## 12. 后续优化

- 使用 Redis Sorted Set 维护实时排行榜。
- 使用分布式锁或 Lua 脚本保证出价原子性。
- 使用客户端 `idempotency_key` 防止重复出价。
- WebSocket 按房间隔离广播，避免不同直播间互相影响。
- 支持断线重连后主动拉取商品详情和排行榜。
- 出价日志异步落库，减轻主链路压力。
