# Base Image Upload Sign Design

## Goal

Allow authenticated clients to upload images directly to Volcengine TOS without sending image bytes through the backend. The backend only issues short-lived upload signatures and returns the final image URL that can be saved into existing fields such as `AuctionItem.ImageURL` and `User.AvatarURL`.

## API

Add the endpoint under the existing `base` module:

```text
POST /api/v1/base/uploads/images/sign
```

The route requires the existing user authentication middleware. Both `user` and `merchant` identities may request signatures.

Request body:

```json
{
  "filename": "cover.png",
  "content_type": "image/png",
  "size": 123456,
  "usage": "item"
}
```

`usage` is used only for object-key grouping in the first version. Allowed values are `item`, `avatar`, and `general`.

Response body:

```json
{
  "upload_url": "https://<bucket>.<endpoint>/",
  "form_fields": {
    "key": "uploads/images/item/user_xxx/2026/06/object_xxx.png",
    "policy": "...",
    "x-tos-algorithm": "...",
    "x-tos-credential": "...",
    "x-tos-date": "...",
    "x-tos-signature": "..."
  },
  "image_url": "https://<public-domain>/uploads/images/item/user_xxx/2026/06/object_xxx.png",
  "object_key": "uploads/images/item/user_xxx/2026/06/object_xxx.png",
  "expires_in": 600
}
```

The client posts a browser `multipart/form-data` upload directly to `upload_url`, including all `form_fields` and the file field required by TOS.

## Validation

The service validates the requested upload before signing:

- `filename` is required and must have an image extension.
- `content_type` must be one of `image/jpeg`, `image/png`, or `image/webp`.
- Allowed extensions are `jpg`, `jpeg`, `png`, and `webp`.
- `size` must be greater than zero and no larger than 10 MiB.
- `usage` must be `item`, `avatar`, or `general`.

The service derives the object extension from the validated filename/content type. It does not trust a caller-provided object key.

## Object Keys

Object keys use a backend-generated layout:

```text
uploads/images/{usage}/{user_id}/{yyyy}/{mm}/{object_id}.{ext}
```

`object_id` should be generated by the backend using the existing project ID style. The path includes the authenticated user ID for auditability and future cleanup jobs.

## Configuration

Add a storage/TOS configuration section to `config.GlobalConfig` and `config.yaml.example`:

```yaml
storage:
  tos:
    enabled: true
    region: cn-beijing
    endpoint: tos-cn-beijing.volces.com
    bucket: live-auction-images
    access_key_id: ""
    secret_access_key: ""
    public_base_url: https://img.example.com
    upload_expires: 10m
    image_max_size_bytes: 10485760
```

Real access keys, bucket names, and production domains must stay out of git-tracked example files. The implementation should fail closed when TOS is disabled or required credentials are missing.

## Architecture

Keep the feature inside `internal/app/base`:

- `base/dto` defines request and response structs.
- `base/service` validates input, generates object keys, and calls an upload signer interface.
- `base/storage` or an equivalent focused package wraps the Volcengine TOS SDK and exposes a small signer interface.
- `base/handler` performs binding checks and renders through `response.OK` / `response.Error`.
- `base/router` registers the authenticated sign route alongside existing health routes.

The service should depend on an interface so unit tests can use a fake signer and avoid network access.

## Error Handling

Service-boundary failures should return `pkg/errorx` errors. Invalid filename, type, size, or usage returns `ErrInvalidRequest`. Missing or disabled TOS configuration returns `ErrInternal` and logs enough context for operators without logging credentials.

## Testing

Unit tests should cover:

- unauthenticated route remains protected through route registration;
- valid requests produce a signed response with expected object-key shape;
- invalid content type, extension, size, and usage fail before calling the signer;
- signer failures are returned as service errors;
- config duration and size fallbacks are deterministic.

Tests must not connect to Volcengine TOS or any external service. Use a fake signer in service tests and fixed time/ID generation where needed.

## Out of Scope

The first version does not store upload records in MySQL, process images, verify that a client completed the upload, delete orphaned objects, or issue STS credentials. Those can be added later if product requirements need upload lifecycle tracking or richer client-side upload behavior.
